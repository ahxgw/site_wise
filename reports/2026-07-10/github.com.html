Context

GitHub is the default home for Git-hosted source code and the developer-facing front of Microsoft's AI strategy. Microsoft paid $7.5B in stock for it in 2018 (deal closed Oct 26, 2018); by July 2024 Nadella told investors Copilot alone was "already a larger business than all of GitHub was when we acquired it." The platform monetizes three ways: per-seat Team/Enterprise subscriptions, GitHub Copilot (moving from flat subscription to usage-based pricing from Jun 1, 2026), and compute metering (Actions minutes, Codespaces). For an index builder, the defining trait is that GitHub's own pages carry almost no crawlable metadata — no JSON-LD, no sitemap (Probe Log) — because the REST/GraphQL API is treated as the canonical structured record, not the rendered HTML. Copilot has also become GitHub's acquisition funnel: 80% of new developers now adopt it within their first week on the platform.

Retrieval Notes (site-specific deltas)

Only where GitHub differs from the default platform assumptions.

Identifier & dedup

owner/repo is the human-facing key, but the immutable ID is the numeric id/node_id — repos can rename or transfer owners while keeping the same ID, so name-based joins break silently on rename. Commits key on SHA (content-addressed, stable); files key on (repo, ref, path), not stable across force-pushes.

Retrieval unit

Four units at different churn rates: repository (near-static metadata), commit (immutable), file-at-ref (mutable pointer), issue/PR (long-lived thread). Flattening "a repo" into one document loses the fact that code, history, and discussion churn independently.

Cross-modal joins

Commit SHA joins commits↔PRs↔CI runs. Repo ID joins issues↔PRs↔releases↔dependency graph. User/org ID joins across repos, gists, and events — logins can change but the numeric ID doesn't.

Authoritative content field

Unlike most sites, GitHub has no "SEO description" competing with the real content — the README is both the human-facing description and the indexable summary. The main distortion vector is star/fork counts, gameable and unsafe as a quality signal without provenance checks.

Freshness & recrawl

Three regimes: commit/push events near-real-time via webhooks/Events API; issue/PR state changes within minutes to hours; repo-level metadata (description, topics, license, default branch) close to static, safe to recrawl weekly.

Quality & spam

Star counts are a known manipulation surface (paid star campaigns, bot-farm stargazers) — the anomalous star deltas on low-profile repos in this report's live search sample (Probe Log) are a visible symptom. Fork count and dependency-graph "used by" counts are harder to fake.

Internal vs external search

GitHub's own code search is the only way to search file contents at scale — generic web search indexes repo metadata and README text but not source lines, since robots.txt disallows crawling blame/raw/commits (Probe Log) and there's no sitemap for the long tail of files.

User query profile

Two intents share one box: "find a library that does X" (discovery, favors stars/topics/README) versus "find where X is implemented" (code search, favors exact/regex match). An index tuned for the first will do badly on the second.

Indexing Plan

  • Retrieval unit: repository as the top-level document, with commit, file-at-ref, and issue/PR as separately-refreshed sub-records rather than flattening everything into one blob.
  • Primary keys & joins: numeric repo id for the repo record; commit SHA for history/CI joins; numeric user/org id for authorship across repos and events.
  • Ingestion paths, ranked: 1) REST/GraphQL API for metadata, issues, releases, dependency graph — 60/5,000/15,000 req/hr depending on auth tier (Probe Log). 2) Git clone (or Contents/Git Data API) for file content — unmetered by the REST quota, the only way to get full text cheaply at scale. 3) GH Archive for historical/bulk event data (stars, pushes, issue opens) back to 2011 without touching GitHub's own quota. 4) GitHub Search API for discovery queries — separate, much tighter budget (10–30 req/min), so best used for targeted lookups, not bulk enumeration.
  • Recrawl cadence: commits/events near-real-time via webhooks where possible; issue/PR state hourly-to-daily for active repos; repo metadata (topics, license, description) weekly; full-text/code content on push, not on a timer.
  • Derivatives to generate: README/code embeddings for semantic code search, star-provenance scoring to down-weight manipulated popularity, a rename-aware ID-to-name resolver so historical links don't 404 silently.
  • Demo to build: a quota-aware GH Archive tailer — stream one hour of public events, bucket by type, and show how much of the graph (new repos, stars, PRs) a free-tier indexer can reconstruct without ever calling the rate-limited REST API.

Risks to the Index

  • No public sitemap, no page-level structured data: the Probe Log's 406 on sitemap.xml and zero JSON-LD on repo/user pages mean any indexer relying on HTML scraping instead of the REST/GraphQL API will silently under-cover the corpus.
  • Search API budget is far tighter than the core API: 10 req/min unauthenticated (Probe Log), so bulk discovery has to ride the Events/Archive paths, not search.
  • Public-repo content is licensed for on-platform viewing and forking, not third-party redistribution. ToS §D.5 grants other users only the right to "use, display, perform and reproduce (by forking) Your Content through the Service" — further reuse rights come solely from whatever OSS license, if any, the owner separately applies. An index treating "public on GitHub" as "freely redistributable" is relying on a right the ToS doesn't grant. GitHub ToS (effective Apr 27, 2026)
  • GitHub now trains on interaction data by default. Since Mar 25, 2026, Copilot Free/Pro/Pro+/Max chat interactions may train GitHub's own AI models unless a user opts out; Business/Enterprise are excluded under their DPA. GitHub Blog policy update
  • GitHub-specific financials appear only in earnings-call remarks, never in SEC filings. Microsoft's FY2025 10-K has no GitHub-specific dollar figure anywhere (confirmed by direct fetch, 2026-07-10) — nothing audited to anchor to beyond Nadella's selectively-disclosed growth percentages.
  • inferred Star-based popularity signals need provenance filtering before use in ranking. Basis: the live search sample in the Probe Log returned repos with star counts inconsistent with their visible profile, a pattern documented industry-wide as paid star manipulation.
