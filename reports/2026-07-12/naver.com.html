Context

Naver Corporation (KRX: 035420) traces to a 1997 in-house project ("Web Glider") at Samsung SDS; founder Lee Hae-jin spun it out as Naver Comm on 1999-06-02, and the naver.com domain itself was registered 1997-09-12 verified (Probe Log WHOIS; Wikipedia). Naver is South Korea's default answer engine and portal, bundling search ads, an internal news-aggregation product, UGC (blog/cafe), commerce (Naver Shopping/Pay), and content (majority-owned, Nasdaq-listed WEBTOON Entertainment) under one login. Q3 2025 revenue was KRW 3.1381T, split Search Platform 1.06T / Commerce 985.5B / Fintech 433.1B / Content 509.3B / Cloud 150.0B verified (NAVER press release). For a full-modal index, Naver is the sharpest available case of a "sovereign portal": the search engine and the content it ranks are the same company, and nearly every owned property's robots.txt explicitly names and blocks AI crawlers — a stronger, more deliberate wall than the generic default.

Retrieval Notes (site-specific deltas)

Only where this site differs from the default platform assumptions. Delete any block that would just restate them.

Identifier & dedup

News article URLs encode a 3-digit press-outlet code plus a per-outlet article ID (/article/{press_code}/{article_id}) verified (Probe Log sample). Both parts are needed to dedup — the same underlying wire story can appear under multiple press codes if more than one partner ran it.

Retrieval unit

For news, the article is the unit, but authorship is external (the press partner), so Naver is a distribution/index layer over content it doesn't own — closer to indexing a search engine's SERP than a publisher's site.

Authoritative content field

Article pages carry Open Graph tags but no JSON-LD/schema.org markup at all verified (Probe Log) — the visible HTML body/headline is the only authoritative field; there is no structured-data shortcut.

Quality & spam

Naver disabled comments sitewide on entertainment-news articles from 2020-02-19 and on sports-news articles from 2020-08-07, following celebrity-suicide controversies tied to malicious comments verified (Korea Herald). A UGC-signal model built on "comments = engagement" will silently see zero comments on two entire verticals, not a low-engagement signal.

Internal vs external search

news./cafe./search./tv.naver.com block essentially all external crawlers, including mainstream search engines on cafe.naver.com (Probe Log) — external search cannot answer Naver-hosted queries at all; a user who wants Naver content must go through Naver's own search or the metered Search API.

Indexing Plan

The actionable conclusion: how Site Wise would build the index for this site.

  • Retrieval unit: news article (press_code + article_id), blog post, cafe post, KnowledgeiN Q&A pair, shopping listing — each a separate object class with its own Search API category.
  • Primary keys & joins: {press_code}/{article_id} for news; Search API result link field as the join key back to the live page for every other category, since no numeric ID is exposed in API responses for blog/cafe/KIN.
  • Ingestion paths, ranked: 1) Search API per category, 25,000 calls/day/category, snippet+link only, no body (free, but a hard ceiling and no full text); 2) direct page fetch of the link URL for body text — legal only insofar as robots.txt allows it, and robots.txt disallows this on every property tested except the www.naver.com homepage itself; 3) becoming an allowlisted UA (Facebook/Twitter-style) for OG-card scraping only — metadata, not body text, and not a documented path for a third party to request.
  • Recrawl cadence: news: minutes (continuous publish observed); blog/cafe: unmeasured, likely daily-to-weekly per post edit; KnowledgeiN: append-only, near-zero mutation of old answers; shopping: price/stock likely sub-daily (unmeasured).
  • Derivatives to generate: none justified without body-text access — Search API snippets are too short for embeddings beyond a title-level signal; a body-text pipeline would require resolving the robots.txt blockage first.
  • Demo to build: a small script that calls the Search API news category for a fixed query on a schedule and diffs link/press_code churn over a day, showing how fast the press-partner mix rotates under one query — the cheapest way to demonstrate the 25,000/day ceiling against real update velocity.

Risks to the Index

Only risks that change indexing strategy: legal/licensing shifts, API deprecations, ownership changes, wall hardening. Skip generic business risks.

  • Every owned property tested names specific AI crawlers (GPTBot, ClaudeBot, Claude-SearchBot, PerplexityBot, Google-Extended, Applebot-Extended, OAI-SearchBot, CCBot, meta-externalagent) in robots.txt and blocks them explicitly rather than by wildcard omission — a deliberate, maintained policy that is unlikely to relax, not an oversight to wait out.
  • cafe.naver.com disallows even Googlebot/Bingbot/Baiduspider/Yandex by name (Probe Log) — mainstream search-engine indexing of Naver's largest forum product is already closed off, so there is no fallback "just use Google's index" path for that modality either.
  • The Search API is the only sanctioned programmatic path and returns snippets, never full body text, at a fixed 25,000-call/day ceiling per category per client ID — any index built on it is structurally a metadata/snippet index, not a full-text one, unless full-page fetch becomes separately licensed.
  • News content is licensed from press partners, not authored by Naver; a change in Naver's press-partnership terms (a recurring point of dispute in Korean media policy) could alter what a snippet is even allowed to show, independent of anything Naver's own API does.

Inferences

  • Naver's own search bot (Yeti) does not ingest blog.naver.com via crawling. Basis: blog.naver.com's robots.txt explicitly disallows User-agent Yeti before the general rules, implying internal/direct-database ingestion rather than HTTP crawl for Naver's own indexing.
  • News comments being absent on a given article is not a low-engagement signal but a categorical policy exclusion for two verticals. Basis: Korea Herald reporting on the 2020-02-19 (entertainment) and 2020-08-07 (sports) comment shutdowns, sitewide by section rather than per-article.
  • The unofficial comment-box endpoint is not intended for third-party use. Basis: Probe Log — a guessed request to apis.naver.com/commentBox was rejected with an "invalid access" error rather than returning data or a documented auth challenge.
