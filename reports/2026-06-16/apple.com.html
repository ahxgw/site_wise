Context

apple.com is two corpora wearing one domain. The visible site is a promotional and retail surface — product pages, the newsroom, support docs, store locator — that Apple deliberately makes easy to index: rich JSON-LD, five public sitemaps, and a robots.txt that blocks no AI crawler (Probe Log). It is thin on data, though: product pages carry no price or offer in their structured data. The asset an index actually wants is Apple's media catalog — apps, music, podcasts, books, TV — surfaced not on apple.com but through the public itunes.apple.com Search/Lookup API, which returns structured records with artwork, previews, and stable numeric IDs without any authentication. The catch is throughput, not access: the API is capped near 20 calls/minute and bulk is pushed to a partner feed. Apple is a hardware company ($416B FY2025) increasingly carried by Services ($109B, first $100B+ year), and that catalog is the Services engine. inferred Apple treats marketing content as free advertising and the catalog as a metered funnel. Basis: robots.txt allows all crawlers including AI agents (observed 2026-06-16) while the catalog API throttles to ~20/min and routes heavy use to the Enterprise Partner Feed (see Verified Metrics).

Retrieval Notes (site-specific deltas)

Only where apple.com differs from the default platform assumptions.

Identifier & dedup

The catalog's universal key is the numeric adam ID (trackId/collectionId/artistId), present in every API record and in music.apple.com/.../{id} and apps.apple.com/.../id{id} URLs. It is storefront-scoped: us, cn, jp are separate namespaces for the same content. The website instead uses path-based locales (/cn/, /jp/) — dedup those against the canonical www.apple.com page.

Retrieval unit

Two unrelated units: the marketing/editorial page (URL) for the website, and the catalog item (adam ID) for apps/music/podcasts/books/TV. They share artwork on mzstatic but almost nothing else.

Cross-modal joins

artistId joins albums↔tracks↔music videos; bundleId↔App ID joins app metadata to its store page; UPC/EAN lookup bridges Apple's namespace to external music IDs. All catalog media joins back to mzstatic artwork URLs.

Authoritative content field

Marketing copy is the least neutral field — promotional by design and the wrong source for factual specs. Catalog API fields (publisher-supplied) and the support docs are the trustworthy text; product spec tables beat the hero copy.

Freshness & recrawl

Website is event-driven: near-static between keynotes, then a large coordinated swap on launch day. Catalog is the opposite — prices, availability, and chart rank move daily, so the API/RSS side needs a far tighter cadence than the crawl side.

AI-crawler stance (reversed default)

Unlike most large platforms, robots.txt blocks no AI crawler (Probe Log). Apple's own opt-out lever, Applebot-Extended, is absent from its file — and Apple's docs say absence = consent to train foundation models. Apple permits AI training on its own marketing content while offering other publishers the switch.

Internal vs external search

Apple runs its own crawler, Applebot, to power Siri/Spotlight/Safari suggestions — so apple.com is both a source and a search engine. The rich JSON-LD shows it still courts Google; the two coexist rather than the site escaping to site:.

Geo & sovereignty split

From a China vantage the site is served by a third-party Chinese CDN (Probe Log), with a geo cookie steering storefront and locale. A single-region crawl silently misses the localized corpus; the catalog's per-storefront IDs make this explicit.

Indexing Plan

  • Retrieval unit: two classes — website page (URL + locale) and catalog item (storefront + adam ID).
  • Primary keys & joins: canonical www.apple.com URL for the site; storefront + adamID for catalog; artistId/bundleId/UPC for cross-item and cross-store joins; mzstatic artwork URL as the shared media key.
  • Ingestion paths, ranked: 1) the 5 public sitemaps — full marketing, newsroom (with video), support, retail; freely crawlable incl. AI. 2) iTunes Search/Lookup API — catalog metadata, artwork, previews; no auth but cache hard against the ~20/min cap. 3) App Store RSS chart feeds — cheap daily freshness/ranking signal. 4) EPF — the only sanctioned bulk catalog path (partner approval). 5) Review text — no official path; design around ratings/counts.
  • Recrawl cadence: marketing pages on keynote/launch events; newsroom daily; catalog charts daily; catalog metadata weekly; retail locations monthly.
  • Derivatives to generate: artwork embeddings for visual dedup, audio fingerprints from previewUrl, spec extraction from product tables, an adam-ID↔UPC↔bundleId concordance.
  • Demo to build: a storefront-aware iTunes Search indexer that honors the ~20/min cap, caches aggressively, emits JSONL keyed on adam ID with artist/app joins, and logs which fields throughput forces you to skip — the music/app analogue of the Amazon PA-API demo.

Risks to the Index

  • The generous API is undocumented-stable: the iTunes Search API has no versioned contract and Apple can change the ~20/min cap "at any time" (docs); bulk depends on EPF partner status that can be revoked.
  • Reviews have no legitimate bulk path: App Store ratings/review text are only on undocumented per-app feeds — plan for ratings-summary-only permanently.
  • inferred The open AI stance is reversible. Basis: robots.txt carries no Applebot-Extended rule today (observed 2026-06-16); Apple already ships that exact opt-out lever, so a one-line addition would cut AI training access to apple.com overnight.
  • Geo-split corpus: China (and other regions) are served by separate CDNs/storefronts; an index crawling from one vantage under-counts localized content and per-storefront catalog IDs.
