Context

x.com is the renamed Twitter: a public posting network where short messages, replies, and reposts form a real-time conversation graph. Elon Musk bought Twitter for $44B in October 2022, rebranded the domain to x.com in July 2023, and in March 2025 folded the company into his AI lab xAI in an all-stock merger that valued X at $33B. That ownership change is the dominant fact for an index builder: X's public posts are now the training and grounding corpus for Grok, so the platform has every incentive to keep its real-time stream inside the xAI/X walls rather than let it be crawled. The product still earns mainly from advertising plus Premium subscriptions, but the data itself is now a strategic AI asset. inferred X treats its post stream as a proprietary AI input, not a crawlable corpus. Basis: robots.txt sets User-agent: * Disallow: / and allowlists only three SEO/social bots (Probe Log), while the same content feeds Grok via the /os-grok.xml OpenSearch descriptor served to Googlebot (Probe Log).

Retrieval Notes (site-specific deltas)

Only where x.com differs from the default platform assumptions.

Identifier & dedup

The post ID is a Snowflake — a 64-bit, time-sortable integer in every URL (/{handle}/status/{id}). It is the strong key. The handle in the path is not stable (users rename; the ID still resolves), so canonicalize on the numeric ID, not the handle. twitter.com and x.com paths are aliases for the same object; t.co wraps every outbound link and must be unwrapped.

Retrieval unit

The post is the unit, but it is rarely meaningful alone: retrieve it with its conversation_id thread and quoted-post chain attached, or short replies lose their referent.

Cross-modal joins

Everything joins on post ID and author ID. Media has no independent stable ID exposed publicly — images/video are reachable only through their parent post, so a deleted post orphans its media. There is no durable key from a pbs.twimg.com URL back to a post.

Authoritative content field

The post body is authoritative; there is no SEO title layer to distort it. The distortion surface is instead volume and coordination — bot networks and reply spam — not keyword stuffing.

Freshness & recrawl

Unusually high mutability: posts are edited (Premium), deleted, or hidden when accounts flip private or are suspended, and the author's visibility (blue-check boost) changes ranking after the fact. An index must re-verify existence, not just re-rank.

Internal vs external search

On-site search owns real-time/recent intent (/search/realtime and /search/users are explicitly robots-blocked, so external engines cannot mirror it). Grok is wired in as a second search descriptor (/os-grok.xml, Probe Log) — conversational queries route to the model, not a results list.

Crawl posture

Sharper than the default two-track model: the generic public-crawl middle is fully closed (* Disallow: /), only three named SEO/social bots get a curated allow-list, and everything else — including AI — is pushed to metered API credits or an Enterprise firehose. Even the declared sitemap 403s to guests.

User query profile

Real-time intent dominates: breaking news, live events, "what are people saying about X right now," brand/handle monitoring, and cashtag/finance chatter. Recency outranks authority more than on most corpora.

Indexing Plan

  • Retrieval unit: the post, keyed by Snowflake ID, stored with its conversation thread, quoted chain, author ID, and media URLs as attached sub-records.
  • Primary keys & joins: post ID (canonical) + author numeric ID; conversation_id for threads; unwrap t.co for outbound-link joins; treat handle as a mutable display alias.
  • Ingestion paths, ranked: 1) Enterprise firehose/filtered stream — the only path that scales to real-time coverage, but priced at ~$42K/mo+. 2) Pay-per-use v2 reads — $0.005/read, 2M/mo cap, fine for targeted monitoring, not corpus building. 3) oEmbed — free, but one URL at a time and display-only. 4) Generic crawl — closed (Probe Log), do not design around it.
  • Recrawl cadence: existence re-checks matter more than re-ranking — sample active conversations continuously; re-verify older posts on a slower sweep because deletion/suspension is common.
  • Derivatives to generate: thread reconstruction from conversation_id, link-unwrapping table for t.co, language ID, and bot/coordination scoring (the dominant spam axis here).
  • Demo to build: an oEmbed-only post-card indexer — paste post URLs, fetch via publish.twitter.com/oembed with no auth, emit JSONL index docs (author, canonical URL, embed HTML, cache_age), and show exactly which fields the no-auth path omits (engagement counts, media variants) versus what the metered API would add.

Risks to the Index

  • Data is now an AI moat, not a crawl target: after the xAI merger (Verified Metrics) the post stream feeds Grok, so expect access to tighten, not loosen — the opposite trajectory of an opening API.
  • Pricing volatility: X moved free→tiered→pay-per-use and migrated Basic subscribers off within ~3 years (Verified Metrics); any cost model built on today's per-request prices can be repriced unilaterally.
  • No legitimate bulk path below Enterprise: the firehose is the only real coverage path and costs ~$42K/mo+; everything cheaper is rate-capped (2M reads/mo) and cannot build a corpus.
  • inferred AI crawlers are explicitly unwelcome. Basis: Google-Extended Disallow: * in robots.txt (Probe Log) and the absence of any published AI-licensing program for third parties — X reserves the corpus for its own model.
