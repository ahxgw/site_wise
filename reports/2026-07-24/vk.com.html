Context

VKontakte (vk.com) is Russia's largest social network, launched by Pavel Durov in October 2006 and now the flagship product of VK IPJSC (MOEX: VKCO), the country's largest consumer internet group. VK's ownership consolidated under state-linked shareholders after 2021 — Sogaz (a Gazprom-linked insurer), Gazprom-Media, and Gazprombank hold their stakes through the joint vehicle MF Technologies — and the holding company redomiciled from a Cyprus/BVI structure to Russia as VK IPJSC in September 2023. inferred VK's product portfolio is treated as separate research targets, not folded into this report. Basis: Odnoklassniki, Dzen, VK Music, VK Video, and the MAX messenger each report distinct MAU figures in VK's own segment disclosure (Verified Metrics) and sit on distinct hosts. VKontakte monetizes primarily through advertising and, increasingly, social commerce (in-post product cards); it sits inside VK Group's "Social platforms and media content" segment alongside those sibling products.

Retrieval Notes (site-specific deltas)

Only where VK differs from the default platform assumptions.

Identifier & dedup

User and community IDs are plain sequential integers; community/group walls use a negative owner_id convention, giving a cheap, stable way to distinguish person vs. group content without a lookup. vk.com and vk.ru resolve the same product (Probe Log: vk.ru sits in vk.com's own CSP allow-list) and should dedup as one canonical host; m.vk.com is a mobile redirect target, not a separate corpus.

Retrieval unit

The wall post (with its inline attachments array) is the natural unit — it already carries photo/video/audio/doc IDs, so a single API call yields the cross-modal join, unlike platforms that require a second lookup per media object.

Cross-modal joins

Attachment IDs inside a wall post are the join key to photos/videos/audio/docs; owner_id + numeric ID is the join key back to the author (user or community). No separate concordance table is needed for the common case.

Authoritative content field

Wall post text is canonical; unlike marketplace listings there's no seller-controlled title field competing with it. Pinned and promoted (VK Ads) posts are algorithmically boosted and should be flagged distinctly from organic reach, since they are over-represented in any casual sample.

Freshness & recrawl

Wall posts and comments mutate and delete frequently; profile/community metadata (name, description, membership) changes slowly. A single recrawl cadence across both wastes budget on the slow-moving half.

Quality & spam

Russian-language social platforms have well-documented bot/engagement-farm activity, but VK has not published a fraud figure comparable to Amazon's fake-review count (Open Questions) — trust scoring here would run without a company-provided baseline.

Internal vs external search

robots.txt names only Googlebot and Yandex by user agent (Probe Log); every other crawler — including other general search engines, not just AI agents — inherits the wildcard block set. External indexing of VK content outside Google/Yandex has no stated allowance at all.

User query profile

Predominantly Russian-language: person/community name lookups, group/content discovery, and messaging-adjacent search. A full-modal index needs Cyrillic-aware tokenization and transliteration handling as a first-class concern, not an edge case.

Indexing Plan

  • Retrieval unit: wall post, with its inline attachments array as fast-refresh sub-records; profile/community record as a slow-refresh parent.
  • Primary keys & joins: owner_id (positive = user, negative = community) + post/object ID; attachment IDs embedded per post for direct media joins.
  • Ingestion paths, ranked: 1) VK API (wall.get, newsfeed.search, photos.get, video.get) via an OAuth-registered app — the only path that returns real content; per-app quota undocumented in this sample (Open Questions). 2) Public crawl — effectively non-functional for a general indexer: robots.txt only names Google and Yandex, and profile/wall pages render as an empty SPA shell for other agents (Probe Log). 3) No sitemap, no oEmbed, no bulk dump or firehose found.
  • Recrawl cadence: wall posts/comments on a short cycle (hours) for active accounts; profile/community metadata weekly; media objects on post edit.
  • Derivatives to generate: Cyrillic-aware language ID and transliteration normalization, OCR for image posts, ASR for VK Video content, engagement-fraud embeddings given the ecosystem's documented bot-farm incentives.
  • Demo to build: a token-scoped VK API wall-post fetcher that resolves owner_id + attachment IDs into a joined JSONL record, surfacing whatever the per-app quota forces you to skip (mirrors the Amazon PA-API demo pattern).

Risks to the Index

  • inferred API access carries geopolitical/sanctions exposure that most platforms in this project don't. Basis: VK's majority voting control sits with Sogaz, Gazprom-Media, and Gazprombank via MF Technologies (Dec 2021 consolidation, Context); a foreign indexer's ability to register or retain a VK developer token is itself subject to sanctions-regime changes, not just VK's own API terms.
  • No anonymous tier exists to lose: every API method sampled required a token (Probe Log) and no oEmbed endpoint was found — there's no free/public surface left to further restrict, unlike platforms mid-lockdown.
  • inferred Cross-host fragmentation risks entity duplication. Basis: vk.ru, m.vk.com, and vkvideo.ru all appear in vk.com's own CSP allow-list (Probe Log) as related-but-distinct hosts; an index that crawls them independently without the owner_id/attachment join keys above will double-count the same underlying objects.
  • Financial fragility: VK Group posted its sixth consecutive annual net loss in FY2025 (Verified Metrics); a business-model shift (e.g., tighter API monetization, deeper state subsidy dependence per reporting on state funding for VK Video) could change API terms with little notice.
