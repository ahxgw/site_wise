Context

open.spotify.com is the browser front end for Spotify, the world's largest audio-streaming service: 761M monthly listeners and 293M paying subscribers across 184 markets (Verified Metrics). The licensed audio itself is DRM-protected and never legitimately bulk-retrievable; the interesting object for an index builder is the catalog graph — tracks, albums, artists, playlists, podcast shows/episodes, audiobooks — each a base-62 ID with stable cross-links. Spotify exposes this graph three ways with sharply different ceilings: every public object page carries rich OpenGraph + JSON-LD music metadata (Probe Log), a no-auth oEmbed endpoint returns an embeddable player and thumbnail, and the OAuth-gated Web API returns structured records. inferred Spotify is deliberately narrowing the machine-readable surface to the embed/search lane while closing the catalog-graph and recommendation lane. Basis: robots.txt blocks training crawlers with the comment "avoid feeding model weights with our catalogue graph," yet explicitly allows /embed/ and /oembed for search/agent fetchers (Probe Log); the Nov 2024 Web API cuts removed audio-features, audio-analysis, recommendations and related-artists (Verified Metrics).

Retrieval Notes (site-specific deltas)

Only where open.spotify.com differs from the default platform assumptions.

Identifier & dedup

Every object is a 22-char base-62 ID, present in the URL (/track/{id}), the spotify:track:{id} URI, and the JSON-LD @id. Dedup against deep-link aliases (spotify://, ?si= share params) and against per-market duplicates — the same recording can have several IDs across regional releases; ISRC is the cross-ID anchor.

Retrieval unit

The track is the atom for music, but playlists and editorial collections are the distribution unit users actually navigate. Podcasts split into show (series) and episode (item) — index both, joined by show ID.

Cross-modal joins

OG/JSON-LD give free joins without the API: music:album and music:musician are emitted as open.spotify.com URLs, so track→album→artist links are extractable from the page head alone. ISRC↔track and UPC↔album (API only) join Spotify's namespace to the wider music-rights graph.

Authoritative content field

Catalog text comes from rights-holder delivery (DDEX), so titles and credits are clean — unlike seller- or user-authored fields elsewhere. The og:audio preview is the only audio leakage outside the licensed player.

Freshness & recrawl

Two regimes: the catalog graph (tracks, albums, credits) is near-immutable once delivered; editorial and algorithmic playlists rotate daily and podcast feeds append continuously. Recrawl playlists/episodes daily, catalog objects monthly.

Quality & spam

Spotify-specific abuse is stream-farming and "fake artist" / AI-generated upload floods, plus metadata gaming (keyword-stuffed titles, fake features). The trust signal lives in Spotify's internal anti-fraud, not in any exposed field.

Internal vs external search

Discovery is overwhelmingly in-app (search box + recommendations). The robots policy keeps Google/Bing to /embed/ and /oembed, so external web search indexes only the embed/share sliver, not the catalog — site:open.spotify.com is shallow by design.

User query profile

Song / artist / podcast name lookups, "play X," mood/activity playlist queries, and lyric-fragment searches (lyrics not in API). High navigational and recommendation intent; little long-tail informational text.

Indexing Plan

  • Retrieval unit: track for music, show + episode for podcasts, audiobook for spoken-word; playlists indexed as collections that reference object IDs.
  • Primary keys & joins: base-62 object ID as PK; ISRC (recording) and UPC (album) for cross-catalog joins; show ID joins episodes; artist/album URLs from OG link the graph without the API.
  • Ingestion paths, ranked: 1) Web API objects + relationships — cleanest structured data, but OAuth, 25-user cap until extended quota, and rolling-30s rate limit; recommendations/audio-features no longer available to new apps. 2) Per-page OG + JSON-LD on the public sliver — no auth, gives title/album/artist/duration/release-date/preview, but only for crawlable pages and within robots limits. 3) oEmbed — no auth, but yields just embed HTML + title + one thumbnail. 4) Full audio — no legitimate bulk path; design the index to be metadata-only.
  • Recrawl cadence: editorial/algorithmic playlists and podcast episodes daily; track/album/artist catalog monthly; cover art on change only.
  • Derivatives to generate: ISRC/UPC concordance to external rights data; artist→album→track graph from OG links; image embeddings for cover-art dedup across regional duplicates. No ASR — full audio is not retrievable.
  • Demo to build: a page-metadata harvester — paste Spotify URLs, pull OG + JSON-LD only (no API key), and emit JSONL catalog-graph docs, showing exactly which fields the public page yields versus what the OAuth API adds.

Risks to the Index

  • The API surface is shrinking, not growing: the Nov 2024 cuts removed audio-features, audio-analysis, recommendations, related-artists and editorial playlists for new apps (blog); plan for metadata-only and assume further endpoints can close.
  • Extended quota is the real gate: a 25-user development-mode cap makes any consumer-scale index depend on a discretionary approval that Spotify can deny or revoke.
  • inferred The public-page lane is the durable one, but explicitly steered to embed/search use. Basis: robots.txt allows search/agent fetchers only /embed/ and /oembed while blocking all named training crawlers at Disallow: / (Probe Log) — an index relying on page OG must present as an allowed fetcher, not a training crawler.
  • Audio is permanently off-limits: full streams are DRM/licensed and the 30s preview is the only leak (via og:audio); any audio-derivative feature must be designed around having no audio corpus.
