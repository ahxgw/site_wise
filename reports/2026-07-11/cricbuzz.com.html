Context

Cricbuzz is a cricket-only news, live-score, and stats portal founded in 2004 by Pankaj Chhaparwal, Piyush Agrawal, and Pravin Hegde verified (Wikipedia). Times Internet, the digital arm of the Times of India Group, bought a majority stake in November 2014 and merged its own GoCricket property in verified (Inc42). In May 2025, Dream Sports — parent of fantasy platform Dream11 — bought a minority stake in Cricbuzz and sister property Willow TV for $50M, aiming to bolt fantasy/e-commerce features onto the score feed verified (SportsPro). Revenue is programmatic/direct display advertising (1,027-line ads.txt, Probe Log) plus the Cricbuzz Plus ad-free subscription and the Cricbuzz11 fantasy product launched February 2025 verified (Wikipedia). For a full-modal index, Cricbuzz matters as the highest-traffic single-sport vertical in cricket's biggest market: dense structured entities (matches, players, teams) updated at match speed, thin on original media of its own.

Retrieval Notes (site-specific deltas)

Only where this site differs from the default platform assumptions. Delete any block that would just restate them.

Identifier & dedup

The numeric ID in the URL path is canonical, not the slug. The same article (ID 139512) appears in the news sitemap under one slug and resolves with a different slug in its own <link rel=canonical>/og:url (plus a -cricbuzzcom suffix appended at render time) inferred Basis: sitemap slug for /cricket-news/139512/ differs from the canonical/og:url slug served on that same page, Probe Log. Dedup on ID, never on slug text.

Retrieval unit

Match (by match ID) is the central entity, joined to four page types — live score, scorecard, match facts, squads — that should be merged into one record rather than indexed as four separate documents.

Authoritative content field

Article permalinks carry no per-article JSON-LD (only site-wide Organization/WPHeader), while the homepage embeds thin NewsArticle/VideoObject stubs (name + image only) for whatever it is currently featuring inferred Basis: Probe Log structured-data sample, homepage vs. article permalink. Neither surface is a substitute for parsing the visible headline/byline HTML directly.

Indexing Plan

The actionable conclusion: how Site Wise would build the index for this site.

  • Retrieval unit: match record (live-score + scorecard + facts + squads merged by match ID) for live coverage; article (by article ID) for news/editorials; player and team profiles as joined entities.
  • Primary keys & joins: numeric IDs embedded in the URL path — match ID, article ID, player-profile ID, team ID — are the only stable keys; slugs decorate the URL but drift (Retrieval Notes).
  • Ingestion paths, ranked: 1) sitemap-driven crawl of the 11 child sitemaps under a browser-like/allowlisted UA (free, but the wildcard robots rule and Akamai's UA gate mean an unnamed crawler gets blocked before it starts, Probe Log); 2) direct page fetch for article/profile/match HTML, which is already server-rendered (no headless browser needed per probe); 3) third-party "unofficial Cricbuzz" API wrappers on RapidAPI that reverse-engineer the same internal /api/ surface robots.txt disallows — usable but rests on an undocumented ToS relationship, not an official grant.
  • Recrawl cadence: live match pages during play (interval unmeasured — Open Questions); news hourly (sitemap changefreq); videos daily; player/team/series pages weekly.
  • Derivatives to generate: embeddings over article text and player/team stat rows for cross-entity search; ASR captions on Kaltura highlight clips (none observed) for accessibility and searchable transcript text — the only original video is short analysis/forecast clips, so ASR cost is low.
  • Demo to build: a small crawler that resolves a batch of news-sitemap URLs, fetches each permalink, and shows the canonical-slug vs. sitemap-slug mismatch alongside the homepage-vs-permalink JSON-LD gap — proving ID-based dedup is required before any slug-keyed pipeline is trusted.

Risks to the Index

Only risks that change indexing strategy: legal/licensing shifts, API deprecations, ownership changes, wall hardening. Skip generic business risks.

  • Akamai Bot Manager blocks non-browser UAs on every path tested, including robots.txt itself (Probe Log) — a tightened bot check could close simple HTTP-client ingestion entirely, forcing headless-browser or partner-feed dependence.
  • The /api/ surface that would give structured match data directly is robots-disallowed and unauthenticated calls redirect (302) rather than returning JSON; the only public wrappers around it are unofficial RapidAPI listings with no visible license from Cricbuzz, a legally fragile dependency for any production pipeline.
  • Video content_loc URLs in the sitemap carry signed Kaltura session tokens of unknown lifetime; a crawl that caches these URLs instead of re-resolving them from the live sitemap will silently rot.
  • Ownership just added a second stakeholder (Dream Sports, 2025-05) with its own fantasy-sports incentives; a future fuller acquisition or content-licensing shift toward Dream11's properties would change what's safe to re-display.

Inferences

  • AI crawlers (GPTBot, ClaudeBot, PerplexityBot, etc.) are blocked by omission, via the catch-all User-agent: * / Disallow: /, not by an explicit named rule. Basis: robots.txt lists a long named allowlist (Googlebot, Bingbot, Yandex, Baidu, Twitterbot, Applebot, AmazonAdBot...) with no AI-crawler names anywhere, and the wildcard block disallows everything not named.
  • The /api/ path is session/auth-backed rather than a public data surface. Basis: robots.txt disallows /api/ and an unauthenticated GET to /api/cricket-match/commentary/129453 returned HTTP 302 instead of JSON.
  • The article-permalink slug is decorative, not canonical. Basis: sitemap slug for article ID 139512 differs from the same page's <link rel=canonical>/og:url slug.
  • Structured metadata is inconsistent by page type: rich (but shallow) on the homepage feed, absent on the permalink. Basis: homepage JSON-LD sample contained 10 NewsArticle + 3 VideoObject stubs; the article's own page had only Organization/WPHeader JSON-LD.
  • Kaltura content_loc playback URLs are time-limited grants, not permanent links. Basis: the URLs embed a Kaltura ks= signed-session parameter, the standard Kaltura pattern for expiring playback tokens.
  • RSS feed infrastructure is dead code rather than an active ingestion path. Basis: all five tested /rss/*.xml paths returned HTTP 200 with content-length: 0 instead of a 404/redirect.

Open Questions

  • Real update interval for a live match page during play — sitemap declares changefreq: weekly for live-matches.xml, which cannot reflect actual in-play cadence. Would need a timed poll of one live match's scorecard page at 30–60s intervals.
  • Photo count per gallery — the sitemap lists only 10 gallery index pages, not individual photos. Would need to open a gallery page and count image objects/pagination.
  • Whether Cricbuzz11 (fantasy) exposes any distinct public player/stat feed. Would need a probe of the Cricbuzz11 app/domain directly.
  • Whether any comment or review surface exists (none found logged-out). Would need a logged-in-session probe or a check of the current mobile app build.
  • True size of the historical player-profile corpus vs. the 255 sitemapped IDs — the site almost certainly serves older/retired-player profile URLs outside the sitemap. Would need sequential ID probing or a stats-page-derived player-ID list.
