Context

Pinterest is a visual discovery and bookmarking platform: users "pin" images that link out to source pages and organize them onto boards. It earns almost entirely from advertising sold against discovery intent — $1,008M in Q1 2026, with 631M monthly active users, though three-quarters of those users (Rest of World) monetize at roughly a sixth of US ARPU. For an index builder the atomic object is the Pin: an image plus an outbound link plus board/topic context — Pinterest is unusual in being a giant curated pointer graph into the rest of the web, not a content silo. Its legitimate read surface is sharply bifurcated. A large SEO surface (tens of millions of pin, board, and "idea" URLs in 91 sitemap indexes) is served only to an explicit allowlist of ~314 named crawlers; verified every unlisted agent — including all AI crawlers — gets Disallow: / (Probe Log). The official API is OAuth-scoped to the caller's own account and ads, not a corpus read. There is no public bulk pin endpoint.

Retrieval Notes (site-specific deltas)

Only where Pinterest differs from the default platform assumptions.

Identifier & dedup

Pin URLs come in two forms: bare numeric ID (/pin/137359857373863947/) and slug-prefixed (/pin/046-ray-ma--44824958780723406/) — canonicalize on the trailing numeric ID. The harder dedup is at the image layer: the same source image is re-pinned millions of times, so visual/perceptual dedup matters more than URL dedup.

Retrieval unit

The useful unit is the pin (image + outbound link + caption), with the board/topic as a curation signal. Unlike most platforms the pin is a pointer: its value is partly the page it links to, so the retrieval unit straddles Pinterest and the destination site.

Cross-modal joins

Joins run image ↔ caption ↔ outbound URL ↔ board ↔ user. The outbound URL is the bridge to the rest of the index — but it is not exposed by oEmbed, so the join key you most want is the one hardest to get legitimately.

Authoritative content field

Captions are user-supplied and SEO-noisy; the image is ground truth and the outbound page is the real "document." Treat the pin caption as a weak annotation over an external object, not as the content itself.

Freshness & recrawl

Sitemaps are tiered by engagement and quality and stamped with batch dates (observed 2026-06-08), so Pinterest itself signals recrawl priority — follow the tier, not a flat cadence. Shopping and trends churn fastest; boards are near-static.

Quality & spam

Spam concentrates on outbound links (affiliate/cloaked destinations) rather than on-pin text. Quality is best judged on the destination page and engagement tier, both of which Pinterest already encodes in its sitemap tiering.

Internal vs external search

Pinterest deliberately courts external search: the allowlist exists to feed Google/Bing a curated SEO surface while blocking everyone else. Much Pinterest discovery actually starts as a Google image/query click into a pin page — the SEO surface is the public product.

User query profile

Queries are inspirational and visual ("kitchen ideas", "fall outfits"), high on intent-to-act and shopping, low on named-entity lookup. Visual/similarity retrieval fits better than keyword retrieval.

Indexing Plan

  • Retrieval unit: the pin (image + caption + outbound URL), with board/topic as curation context and the destination page joined in from the rest of the index.
  • Primary keys & joins: numeric pin ID (canonical); secondary keys board ID, user handle, and the outbound URL as the bridge to non-Pinterest documents.
  • Ingestion paths, ranked: 1) Allowlisted SEO crawl of the 91 sitemaps — the only path to pins at scale, and it requires bot-submission approval (Probe Log); honor the engagement/quality tiers as a priority queue. 2) oEmbed per pin URL — unauthenticated, but only title/author/thumbnail, no body or outbound link. 3) API v5 — rich, but account-scoped (own pins/boards/ads), not a corpus read. 4) Generic crawl — blocked by User-agent: * Disallow: /.
  • Recrawl cadence: drive by sitemap tier and batch date, not a flat interval — tier-0 high-engagement first; boards monthly; shopping/trends fast.
  • Derivatives to generate: image embeddings for perceptual dedup of re-pins, caption-language normalization, and an outbound-link resolver that joins each pin to its destination document.
  • Demo to build: a sitemap-tier-walker — read the 91 sitemap indexes, bucket pin URLs by engagement/quality tier and batch date, then enrich a sample via oEmbed and report exactly which fields (outbound link, board, body) the public surface forces you to leave empty.

Risks to the Index

  • Access is an allowlist, not a license: all bulk pin access depends on being one of the ~314 approved UAs; an index that isn't allowlisted has no compliant path to the corpus (Verified Metrics).
  • inferred AI retrieval is blocked by default, with no opt-in. Basis: no AI crawler is named in the 1,045-line robots.txt, so all are caught by the terminal User-agent: * Disallow: / (observed 2026-06-14); inclusion requires manual bot-submission review.
  • The outbound link graph — the most valuable join key — has no public path: oEmbed omits it and the API only returns your own pins, so a key index feature can't be built compliantly at scale.
  • Client-rendered pages with no JSON-LD: the SEO surface relies on OG tags plus JS rendering; a parser that doesn't execute JS or isn't served the allowlisted render gets a near-empty shell (Probe Log).

  • Outbound-link dead/decay rate — measurable by resolving a sampled set of pin destination URLs over time.
