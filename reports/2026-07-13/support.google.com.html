Context

support.google.com is Google's unified Help Center: one host serving product-specific documentation and moderated community Q&A for essentially every consumer and business Google product (Accounts, Mail, Drive, Docs, Chrome, Android, YouTube, Ads, Merchant Center, Workspace admin, and more). It carries no separate business model of its own — it is a cost center that reduces paid-support load and supports retention across Google's ad-, subscription-, and cloud-revenue products. verified The domain has resolved since at least Feb 2002 (earliest Wayback Machine capture, a 302 redirect), well before the current per-product Help Center structure existed. inferred Content is organized as two distinct object types sharing one host and product-scoped ID space: authored help articles (/{product}/answer/{id}) and community-authored Q&A threads (/{product}/thread/{id}) with algorithmically flagged "Recommended Answers". Basis: probes below confirm both URL families resolve under the same product prefixes and thread pages embed an automated recommended-answer classifier string.

Retrieval Notes (site-specific deltas)

Only where this site differs from the default platform assumptions.

Identifier & dedup

The stable ID is a compound key: (product prefix, answer or thread ID), e.g. accounts/answer/27441. The same numeric ID is reused across unrelated products (Probe Log: mail/answer/27441 404s), so a global numeric index without the product prefix will silently collide.

Retrieval unit

Two distinct units on one host: the article (authored, single accepted answer) and the thread (community, many replies plus one algorithmically flagged Recommended Answer). Treating both as the same document type loses the trust-tier signal that only threads carry.

Cross-modal joins

Locale variants of the same article join on (product, answer ID) with hl stripped — but there is no hreflang markup to discover siblings; an indexer must enumerate ?hl= values itself. Embedded videos join to YouTube via the video ID inside the JS-injected VideoObject, which requires rendering, not just fetching, the page.

Authoritative content field

Article body text is server-rendered and authoritative for facts; the QAPage/VideoObject JSON-LD is a derived, client-only annotation of the same content, not an independent source — don't index it as if it were separately authored.

Freshness & recrawl

Articles: silent edits on product changes, no visible last-modified date on the sampled page — recrawl on a fixed schedule since there's no changed-since signal. Threads: reply-driven, open-ended; the Recommended Answer flag can flip after recrawl as new replies arrive.

Quality & spam

Community answers carry two built-in trust signals absent from generic forums: the algorithmic Recommended Answer flag (explicitly described in-page as automated, not purely vote-based) and the five-tier Product Expert badge (Bronze auto-granted, Gold+ requires Community Manager promotion) — both extractable from rendered thread HTML and usable as a ranking feature without separate moderation infrastructure.

Internal vs external search

robots.txt blocks the on-site search results path (/*/search) for all crawlers, so external indexes cannot mirror Google's own on-site search ranking — they must crawl and rank answer/thread pages independently rather than reusing Google's relevance signal.

User query profile

Troubleshooting and how-to intent tied to a specific product ("gmail not syncing android"), distinct from Amazon/YouTube-style discovery queries — high value for a support/FAQ-oriented retrieval slice of a general index, low value for entity or media search.

Indexing Plan

  • Retrieval unit: article and thread as separate document types, each keyed by (product prefix, numeric ID); locale variants attached as sub-records, not separate documents.
  • Primary keys & joins: (product, id, type) as the primary key; hl enumerated separately per locale since no hreflang exists to discover it automatically; embedded videos joined via YouTube video ID extracted post-render.
  • Ingestion paths, ranked: 1) Direct HTTP GET of answer/thread pages — full text available with no JS, no auth, no documented rate limit encountered in this probe run, but also no bulk listing endpoint, so URLs must be discovered externally (search-engine indexes, product cross-links, or a seed list per product). 2) Headless render — only path to the QAPage/VideoObject schema and any client-fetched supplementary data. 3) No API track exists for this host, unlike Amazon/YouTube-style platforms — there is nothing to apply for.
  • Recrawl cadence: articles monthly (no changed-since signal observed, so cadence is a guess against typical help-doc edit rates); active threads weekly while reply count is climbing, then quarterly once quiet.
  • Derivatives to generate: per-product FAQ clusters from article titles; Recommended-Answer + Product-Expert-tier as a pre-computed trust score for thread replies; locale-normalized canonical article grouping (dedup the ?hl= variants Google itself treats as separate pages).
  • Demo to build: a small crawler that fetches a seed list of answer/thread URLs, diffs plain-HTTP output against a headless-rendered one, and reports exactly which fields (schema.org type, video ID, Recommended-Answer flag) are only recoverable after JS execution — makes the SSR/CSR split in the Probe Log concrete and reproducible.

Risks to the Index

  • No bulk discovery path: with no sitemap, no llms.txt, and no Content API, the only way to discover the full article/thread set is external (search-engine backlinks, product cross-links, or manual seed lists) — coverage is structurally partial, not a temporary gap.
  • inferred Structured-data extraction requires a headless renderer as a hard dependency, not an optimization. Basis: zero JSON-LD found in raw HTML for both sampled page types (Probe Log); the schema is built by a JS function that runs only in a DOM environment.
  • No AI-crawler-specific policy: unlike Amazon and other major platforms, this host's robots.txt names no AI agents at all (Probe Log) — a future policy change here would be a first-seen signal, not an incremental tightening, since there's currently nothing to tighten from.
