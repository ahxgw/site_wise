Context

Indeed is a job-search engine and recruitment marketplace: job seekers search and apply for free, employers pay to post, sponsor, or query candidate data. Launched November 2004 in Austin, TX and Stamford, CT by Paul Forster and Rony Kahan, it overtook Monster.com as the top US job site by October 2010, then was acquired by Japan's Recruit Holdings in October 2012 and run as an independent unit alongside sister company Glassdoor inside Recruit's HR Technology segment. For an index builder, Indeed's three retrievable object types are the job posting (keyed by a jk hash, the most commercially guarded page on the site), the company profile (reviews, salaries, Q&A under /cmp/), and Indeed's own derived labor-market index (aggregated postings counts, published as an open dataset). Only the third has a durable, legitimate bulk-ingestion path; the first two exist only as individually rendered pages behind a Cloudflare bot-check that blocks even a plain, robots.txt-compliant fetch.

Retrieval Notes (site-specific deltas)

Only where Indeed differs from the default platform assumptions.

Identifier & dedup

The job key jk (a hex string, e.g. jk=5b0eeefaea870e59) in /viewjob?jk= is the stable per-posting ID; the company slug in /cmp/{slug} is the stable employer-profile ID. Both are opaque and not otherwise derivable from a search-results listing without following the link.

Retrieval unit

inferred Indeed treats the individual job-detail page as its most commercially guarded surface, more so than search-results or company pages. Basis: /viewjob is disallowed in all three robots.txt UA tiers (Probe Log) — the only path blocked universally — and a direct curl on a live /viewjob URL returned a Cloudflare "Security Check" 403 even though the URL is a plain GET with no query beyond jk.

Cross-modal joins

jk is the join key for a posting; the employer name/slug joins a posting to its company profile, reviews, and salary pages. No numeric company ID was observed in this probe round, so a slug rename would silently break that join — see Open Questions.

Freshness & recrawl

Default assumption of a sitemap-driven recrawl signal does not hold here: both /sitemap.xml and /sitemap_index.xml 404 (Probe Log). The only object class with a genuine, dated freshness signal is the Hiring Lab index (daily rows, weekly-refreshed repo); job and company pages carry no discoverable lastmod equivalent.

Internal vs external search

Opposite of the typical walled-portal pattern: robots.txt explicitly leaves bare /jobs (Indeed's own search-results URL) open to Googlebot, Bingbot, and the named answer-engine tier (Probe Log) — Indeed wants its own SERP-style pages indexed, unlike sites that disallow query-string search pages outright. /rss and /*?rss are explicitly disallowed for every UA tier, so a once-available RSS ingestion path is now closed by policy, not by absence.

Indexing Plan

  • Retrieval unit: one job posting keyed by jk; one company entity keyed by /cmp/ slug (reviews, salaries, Q&A as sub-records); one labor-index row keyed by (country, date, sector) from the Hiring Lab dataset.
  • Primary keys & joins: jk (job); company slug (job → employer profile → reviews → salaries, secondary key); (country_code, date, sector) for the index time series.
  • Ingestion paths, ranked: 1) Hiring Lab GitHub CSV — public GET, CC BY 4.0, no auth, weekly refresh; covers only the aggregate labor-index modality, not individual postings. 2) Enterprise data-partnership program — NDA, sales-led, terms undisclosed open. 3) Per-page fetch of job/company records — technically allowed by robots.txt for major UAs but defeated in practice by a Cloudflare bot-check that 403s even a compliant plain GET; not a legitimate scalable path without a commercial agreement.
  • Recrawl cadence: Hiring Lab index — daily granularity, weekly repo refresh (README, Probe Log). Job/company pages — no sanctioned recrawl path exists at all given the bot-check gate on path 3.
  • Derivatives to generate: none currently justified beyond the index CSVs themselves — no video/audio modality was found to justify ASR/OCR, and job/company text has no bulk path to derive embeddings from without a partnership.
  • Demo to build: a script that pulls the Hiring Lab US aggregate CSV and charts the seasonally-adjusted postings-index trend — legitimate (public GET, CC BY 4.0), and it demonstrates the one ingestion path on this site that survives past the bot-check wall.

Risks to the Index

  • Publisher API and XML feed retirement (2023–2024): the only sanctioned bulk path to job-posting data was shut down; the sole replacement is a sales-led enterprise data partnership with undisclosed terms, so third-party posting-level ingestion has no public, self-serve route.
  • robots.txt's answer-engine allowance is a policy choice, not a technical guarantee, and is revocable at will: the named tier (Claude-User, OAI-SearchBot, PerplexityBot, etc., Probe Log) currently gets broader access than AI-training crawlers, but nothing prevents Indeed from moving any of those UAs into the restrictive tier in a future robots.txt update.
  • Bot-check defeats robots.txt compliance as a practical matter: even paths robots.txt allows for generic UAs (bare /jobs, homepage) 403'd on a plain, policy-compliant curl in this probe round — meaning robots.txt analysis alone overstates what is actually fetchable without a rendering/residential-proxy layer.
  • No sitemap, ever: both sitemap endpoints 404 (Probe Log); there is no lower bound on live posting count from the site itself, so coverage-ceiling estimates for any crawl attempt are unverifiable from public signals alone.
