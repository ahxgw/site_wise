Context

Gemini is Google's conversational AI assistant, launched as Bard on 2023-03-21 and renamed Gemini on 2024-02-08, when this host replaced bard.google.com. Owner is Google (Alphabet); the model is monetized three ways: free app usage feeding the wider ecosystem, consumer subscriptions (Google AI Plus / Pro / Ultra), and a separately hosted metered token API (generativelanguage.googleapis.com, not covered here). For an index builder the decisive fact is a two-domain split: gemini.google.com is a pure logged-in app shell, while the crawlable corpus lives on the separate registrable domain gemini.google — a marketing site with open robots and a 4-sitemap index (Probe Log). Where chatgpt.com hides everything behind a Cloudflare transport challenge, Google's approach is quieter: the app host serves HTTP 200 shells with zero server-rendered text, has no robots.txt at all, and marks the one public per-object surface — /share/<id> conversation pages — noindex, nofollow. inferred The legitimately indexable Gemini corpus is the marketing domain, not the app host. Basis: robots.txt 404, HEAD 204, and a 770KB app shell with no extractable text on gemini.google.com, versus Allow: / plus a 4,165-URL sitemap on gemini.google (Probe Log).

Retrieval Notes (site-specific deltas)

Policy lives in meta tags, not robots.txt

The default assumption is robots.txt AI-crawler blocks; here neither Gemini host blocks anyone at the robots layer (404 and Allow: /, Probe Log). Control is enforced downstream: IP-reputation abuse interstitials at transport, client-side rendering for content, and per-page-class noindex for the only public objects. A robots-only compliance check reads this site as wide open and would be wrong.

Identifier & dedup

Two registrable domains split app and corpus: canonicalize bard.google.com and g.co/gemini/share/* onto gemini.google.com, and strip country dir + hl param on gemini.google to collapse 4,165 sitemap URLs onto ~29 canonical pages. Share IDs are 12-hex tokens — the only stable per-object IDs on the app host.

Authoritative content field

Share-page OG metadata is deliberately generic ("Created with Gemini") — title and description carry no conversation signal, so the only authoritative field is the JS-rendered transcript itself. Marketing pages invert this: JSON-LD (VideoObject) is richer than the visible copy.

Freshness & recrawl

Geo-adaptive serving (French page to a French IP with no locale in the URL, Probe Log) means recrawls from different exits produce spurious diffs; pin crawl geography or diff per-locale URLs only.

Indexing Plan

The actionable conclusion: how Site Wise would build the index for this site.

  • Retrieval unit: marketing page (bare-path canonical) on gemini.google; shared conversation only as an on-demand, user-supplied-link fetch, never as a bulk-indexed unit.
  • Primary keys & joins: canonical bare path for marketing pages, joined to locale variants by path+hl; 12-hex share ID for conversations; VideoObject JSON-LD joins marketing video metadata to page text.
  • Ingestion paths, ranked: 1) gemini.google sitemap crawl — free, open robots, ceiling is the ~29-page marketing corpus; 2) JS-rendered fetch of individual share links on user demand — works (page is public), but noindex forbids putting results in a public index, so treat as retrieval-time-only; 3) nothing else — conversations, Gems, and generated media have no bulk path; don't budget for them.
  • Recrawl cadence: marketing pages weekly, tightening around Google launch events (I/O in May, model launches in Nov); sitemap diff monthly for new feature pages.
  • Derivatives to generate: page-text embeddings for the ~29 canonical pages; harvest VideoObject JSON-LD instead of running ASR on marketing videos. Nothing else is justified.
  • Demo to build: a canonicalizer that crawls all 4,165 sitemap URLs, content-hashes the rendered text, and proves the corpus collapses to ~29 documents — a measurable test of the locale-dedup rule this report asserts.

Risks to the Index

  • Share pages are publicly fetchable but explicitly noindexed — indexing them anyway reproduces OpenAI's Aug 2025 shared-chat privacy incident as a deliberate policy violation, not an accident. Google pre-empted the failure mode; a compliant index gets zero shared-conversation coverage by design.
  • The /sorry abuse gate flipped between 302 and 200 for the same URL within one hour of probing — coverage from datacenter IPs is unstable today and could harden to a Cloudflare-style absolute gate without notice.
  • Host churn is proven, not hypothetical: bard.google.com → gemini.google.com (2024-02), and the Assistant continues merging into Gemini — alias tables need active maintenance or joins silently break.
  • Workspace admins can disable conversation sharing org-wide (Verified Metrics), so enterprise share links can die in bulk from a single admin action — any cached share-link fetch has an unbounded revocation rate.
  • An index scoped to gemini.google.com (the ranked host) sees almost nothing; the corpus sits on gemini.google. Domain-scoping logic that treats them as unrelated registrable domains silently drops the entire indexable surface.

Inferences

  • The legitimately indexable Gemini corpus is the marketing domain, not the app host. Basis: robots.txt 404, HEAD 204, and zero extractable text in the app shell on gemini.google.com, versus open robots and a 4,165-URL sitemap on gemini.google (Probe Log).
  • Google deliberately excludes shared conversations from search engines. Basis: noindex, nofollow meta on the sampled share page, deliberately generic OG description, and a site: search returning a single generic-titled result (Probe Log).
  • The abuse interstitial is IP-reputation-based, not path-based. Basis: the same URL returned 302 → /sorry and then 200 within an hour from the same datacenter exit, while robots.txt fetches were never intercepted (Probe Log).
  • Locale sitemap entries are boilerplate duplicates of the ~29 English pages. Basis: child sitemaps 2–4 repeat sitemap 1's path set under country prefixes with hl params (Probe Log).
