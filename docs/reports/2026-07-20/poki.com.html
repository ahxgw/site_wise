Context

Poki is a free, no-login, no-download HTML5 game portal: pick a game, it plays in the browser. Poki B.V. (Amsterdam) licenses third-party-developed games rather than building them; the co-founders describe poki.com's 2014 launch growing out of a personal web-game hobby project dating to the early 2000s. The company is self-funded (no external investment) and monetizes entirely through advertising shown around gameplay, splitting that ad revenue with the developer who made the game. For an index builder, the interesting object is not the page but the game: a licensed, third-party VideoGame entity with its own developer, rating, thumbnail, and preview clip, wrapped in a templated SSR page that repeats the same marketing boilerplate across roughly 1,480 titles. Poki itself owns none of the games — it is a distribution and ad-monetization layer, closer to an app store than a media publisher, and each title's "exclusive" deal terms (see Retrieval Notes) determine whether poki.com is the only legitimate place to observe it.

Retrieval Notes (site-specific deltas)

Only where Poki differs from the default platform assumptions.

Identifier & dedup

The stable ID is the game slug (/g/{slug}), not a numeric ID. The same game exists under 34 locale prefixes (/en/g/..., /ru/g/..., etc.) with translated title/description but the same slug and asset — dedup by slug across locales, treating locale as a language attribute of one entity rather than 34 separate objects.

Retrieval unit

The game entity (author, rating, thumbnail, preview clip), not the page body. inferred Page prose is templated boilerplate, not distinguishing content. Basis: the sampled game page and category page both use the identical sentence pattern "Play/Discover ... on the most popular website for free online games! ... Play now!" in their meta description.

Cross-modal joins

Slug joins the JSON-LD entity, og:image thumbnail, and og:video clip. No numeric object ID is exposed anywhere observed, so a slug rename (common if a title is rebranded) would silently break the join — see Open Questions.

Authoritative content field

inferred JSON-LD fields (author, rating, dates) are more trustworthy than page prose. Basis: prose is templated marketing copy (see Retrieval unit above), while author/rating/date fields vary per game and match the specific title sampled.

Freshness & recrawl

dateModified is a genuine per-object signal (rare among templated portals) and should drive recrawl priority over the sitewide sitemap lastmod, which restamps to the fetch date for every child sitemap regardless of whether any game inside changed (Probe Log: all four en sitemaps stamped 2026-07-20).

Internal vs external search

robots.txt disallows /*? (query strings), blocking crawl of on-site search-result pages. The only legitimate discovery path is the category taxonomy (147 tags) plus the sitemap, not query replay — a delta from portals that expose crawlable search URLs.

Indexing Plan

  • Retrieval unit: one game entity per canonical slug, with a per-locale text/language sub-record.
  • Primary keys & joins: game slug as primary key; author/studio name as a secondary key to roll games up into a developer entity; category breadcrumb as a many-to-many tag join.
  • Ingestion paths, ranked: 1) Sitemap census + per-URL JSON-LD scrape — self-serve, no auth, public GET (crawl en fully, sample other locales only for translation deltas since the catalog is shared). 2) og:image/og:video CDN URLs for thumbnail and preview-clip capture (Cloudflare Image Resizing params are directly controllable in the URL). 3) No official bulk catalog API for third parties — api.poki.com is undocumented and developers.poki.com is a publishing portal, not a query API — open.
  • Recrawl cadence: daily sitemap diff to catch new slugs (227 added in 2025, ~1 every 1.6 days); per-object dateModified check to prioritize re-fetch over the sitewide lastmod stamp; category taxonomy monthly (147 tags, low churn).
  • Derivatives to generate: thumbnail/preview-clip embeddings for genre clustering, since categories are editorially assigned and won't catch cross-genre or trend-driven titles; OCR on thumbnails is low-value here since titles are in the text field already, not baked into the image.
  • Demo to build: a small crawler that walks en/sitemaps/games.xml, fetches each page's embedded JSON-LD over plain HTTP, and renders a local ratingValue/ratingCount leaderboard across the ~1,480-game catalog — legitimate (public GET, no auth) and demonstrates the exact slug → JSON-LD → thumbnail/video join an index would use.

Risks to the Index

  • Web-exclusivity is the default developer deal: a 5-year exclusive term (SDK docs) means poki.com is often the sole legitimate source for a given title on the open web — if Poki delists it, there is no secondary official copy to fall back on.
  • inferred The SSR surface could be replaced by a client-only runtime. Basis: pages currently render fully via plain curl (no JS needed), but the same pages preconnect to games.poki.com/gdn.poki.com/poki-gdn.com for the actual game bundle — a heavier client-side app shell is architecturally already half-present.
  • Cache/no-store mismatch: the observed cache-control: no-store + cf-cache-status: HIT combination (Probe Log) means HTTP caching headers can't be trusted to signal freshness; recrawl scheduling should key off dateModified, not response headers.
  • No public catalog API: the only legitimate bulk path is sitemap + embedded JSON-LD; if Poki moves game metadata behind api.poki.com auth (already returning 404 with CORS headers, suggesting active development), that path could close without notice.
