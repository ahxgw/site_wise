Context

WhatsApp is Meta's cross-platform messenger, acquired in 2014 and now the largest one on Earth by users. For an index builder it is the clearest example of a top-50 site with almost no public corpus: person-to-person and group messages, calls, media, and statuses are end-to-end encrypted with the Signal Protocol, so Meta itself cannot read them and nothing is crawlable by design. The domain whatsapp.com serves three thin public things instead — a marketing site (features, download, security whitepaper), the FAQ/help center on faq.whatsapp.com, and the web.whatsapp.com client shell, which is a login gate, not content. Monetization does not touch the message stream: WhatsApp makes money through the Business Platform (Cloud API on graph.facebook.com), where companies pay per template message. Channels and Communities add one-to-many broadcast, the only place any WhatsApp content is semi-public, and even that is gated behind the app rather than the web. inferred There is no legitimate bulk path to WhatsApp message content for any index. Basis: E2E encryption on all conversations (security whitepaper) plus the absence of any content API on the Business Platform — it sends messages, it does not read the graph.

Retrieval Notes (site-specific deltas)

Only where WhatsApp differs from the default platform assumptions. The big delta: the methodology assumes a curated SEO sliver plus metered APIs sit over a real corpus — here the corpus itself is cryptographically absent.

Identifier & dedup

The primary identity is a phone number (E.164), not a URL or account handle — unusual for a top-50 site. There are no per-message URLs to dedup because messages have no web addresses. Only faq.whatsapp.com articles and wa.me/<number> deep links have canonical URLs.

Retrieval unit

For anything indexable, the unit is a help article or marketing page. The conversational corpus has no retrieval unit at all — there is nothing to store.

Cross-modal joins

No cross-modal joins exist externally: text, media, and calls live only on end devices. The only join available is business-profile → catalog, and only for a business's own data via the Cloud API.

Authoritative content field

For the help center, the article body is authoritative; OG/title tags mirror it. There is no title/transcript/OCR disagreement to resolve because there is no media pipeline to OCR.

Freshness & recrawl

Help/marketing content is slow-changing (product-release cadence). The message stream is the opposite extreme — infinitely fresh and permanently inaccessible; recrawl cadence is undefined because there is nothing to recrawl.

Quality & spam

Spam is a delivery problem, not a ranking one: no public ranking surface exists to game. The manipulation surface is the Business Platform (unsolicited template blasts), policed by Meta's per-number quality rating, not by any index.

Internal vs external search

In-app search over one's own chats is on-device. External search only ever surfaces faq.whatsapp.com and marketing pages; users never reach WhatsApp content through Google because there is none to reach.

User query profile

Public queries about WhatsApp are how-to and troubleshooting ("how to back up chats", "WhatsApp Web QR"), answered by the FAQ — not content queries. There is no user-generated-content query intent to serve.

Indexing Plan

  • Retrieval unit: help-center article (primary) and marketing page (secondary). No conversational unit — accept that the message corpus is out of scope permanently.
  • Primary keys & joins: canonical FAQ article URL + locale; wa.me number for deep-link entities. Phone number is the identity but is not a public index key.
  • Ingestion paths, ranked: 1) Crawl faq.whatsapp.com — it self-declares index,follow and is the one indexable surface (render JS; plain curl gets a 400/error page). 2) Marketing pages via a headless browser (scripted curl is refused; the declared sitemap is 403-walled per Probe Log). 3) Business Platform docs on developers.facebook.com for structured pricing/capability facts. 4) Message content — no path; do not plan for it.
  • Recrawl cadence: FAQ and marketing weekly to monthly (release-driven); pricing docs on change.
  • Derivatives to generate: locale-clustering of FAQ articles, a capability/pricing table distilled from Business Platform docs. No ASR/OCR — no accessible media.
  • Demo to build: a "public-surface mapper" that, given whatsapp.com, classifies every reachable URL into {crawlable FAQ, browser-only marketing, WAF-blocked, encrypted-and-absent} — a concrete illustration that a top-50 host can be ~99% unindexable.

Risks to the Index

  • The corpus is cryptographically closed, not policy-closed: unlike API-gated sites, no future partnership or paid tier can open message content — E2E encryption means Meta itself lacks plaintext. An index strategy must treat this as permanent, not pending.
  • Marketing surface actively rejects automation: homepage 200 but deeper paths 400 to curl, and the declared sitemap is 403-walled (Probe Log). Any ingestion needs a real browser, and that behavior can harden further.
  • inferred AI crawlers are unwelcome even on the thin public surface. Basis: robots.txt sets Disallow: / for GPTBot, ClaudeBot, Google-Extended, PerplexityBot, Amazonbot and ~10 others while allowing only Meta's own facebookexternalhit (observed 2026-07-03).
  • Business Platform economics and architecture shift under you: pricing moved from conversation-based to per-message on Jul 1 2025 (with a further regional rate change dated Oct 1 2026), and the On-Premises API was deprecated in Oct 2025 leaving Cloud API as the sole path. Any product built on send-side costs is exposed to Meta's category re-pricing.
