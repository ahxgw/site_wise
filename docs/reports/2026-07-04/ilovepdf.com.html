Context

iLovePDF is a Barcelona web utility (founded 2010, CEO Marco Grossi) that runs ~30 browser-based PDF and image operations — merge, split, compress, convert, OCR, sign, plus newer AI tools (summarize, translate, extract). It ranks among the world's most-visited sites on document intent alone. The business is freemium: free anonymous use with file/size caps, Premium subscriptions, and a metered developer API (iLoveAPI) that sells credits. For an index builder this site is the anti-corpus case: the "objects" are users' own private documents, uploaded, processed, and deleted within hours (see Probe Log / Verified Metrics). There is no persistent user-content graph to crawl — no posts, no profiles, no back catalog. inferred iLovePDF's entire crawlable surface is its own SEO landing pages, not user data. Basis: the sitemap enumerates 1,415 URLs that are all tool/marketing/blog pages (Probe Log), and robots.txt disallows only /upload/ — the transient file path — while allowing everything else.

Retrieval Notes (site-specific deltas)

Only where iLovePDF differs from the default platform assumptions.

There is no content corpus

The default assumptions all presume a persistent object graph (posts, media, reviews). iLovePDF has none: the high-value data — user documents — is intentionally destroyed within hours. An index of this site indexes its capabilities and landing pages, never its throughput.

Retrieval unit

The tool page (function), keyed by slug + language. The useful retrieval intent is "which tool does X" and "how do I do X to a PDF", not "find document Y". Treat each tool as an entity with {operation, input/output formats, constraints, credit cost}.

Identifier & dedup

Canonical ID = English slug (merge_pdf, compress_pdf). 26 language variants of each page share one canonical tool; collapse them via the slug, keep language as a facet. og:url gives the canonical per page.

Metadata is OG-only

No JSON-LD / schema.org SoftwareApplication markup (Probe Log), so tool attributes (formats, limits, price) must be parsed from prose or pulled from the iLoveAPI docs — they are not machine-declared on the page.

Crawl policy is wide open

Unusual for a top-50 site: robots.txt blocks only /upload/ and names no AI crawlers. There is little to protect because the crawlable surface is pure marketing; the asset worth walling (user files) is not URL-addressable.

User query profile

Transactional, tool-seeking: "compress pdf", "pdf to word", "merge pdf online". Users arrive from search to perform one action and leave. Retrieval here competes as a how-to/tool answer, not as a document search.

Indexing Plan

  • Retrieval unit: the tool (operation) as an entity; secondary unit = blog/help article for how-to queries.
  • Primary keys & joins: English slug as canonical tool ID; language as a facet; join the sitemap page to its iLoveAPI endpoint (same operation) for {formats, limits, credit cost}.
  • Ingestion paths, ranked: 1) Public crawl of the 1,415-URL sitemap — complete, unrestricted, cheap. 2) iLoveAPI docs — to enrich each tool with structured capability data absent from the pages. 3) User documents — no path, ever; deleted by design.
  • Recrawl cadence: monthly is ample (near-static marketing pages despite the sitemap's blanket daily claim); trigger an out-of-band recrawl when a new tool slug appears (AI tools are the growth edge).
  • Derivatives to generate: a normalized tool/capability table (operation × input formats × output formats × constraints × credit cost); embeddings over tool descriptions for "which tool does X" matching.
  • Demo to build: a sitemap-to-capability extractor — crawl the flat sitemap, collapse 26 language variants to canonical slugs, and emit one JSONL row per tool with its OG metadata plus API-derived attributes; surface how much is left open because no schema.org is published.

Risks to the Index

  • The corpus you'd want doesn't exist and won't: deletion-within-hours is a privacy feature, not a wall that might open. Do not architect around ever getting user documents.
  • inferred Tool inventory is the only volatile axis. Basis: new AI tools (translate-pdf, pdf-summarize, pdf-extract) already appear in the sitemap and homepage nav (Probe Log); the page set otherwise looks stable, so index churn tracks the product roadmap, not content.
  • No structured markup to depend on: if iLovePDF never ships schema.org (none today, Probe Log), capability data stays a parsing/enrichment problem rather than a feed.
  • Open crawl could tighten: the permissive robots.txt is cheap to reverse; a future AI-crawler block would still cost nothing to the index since the surface is only marketing copy.
