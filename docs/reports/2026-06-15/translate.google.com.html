Context

translate.google.com is Google's free machine-translation tool: paste or speak text, point a camera, or drop a document, and it returns a translation across 243 languages. It is the consumer face of the same neural and LLM models Google sells through the Cloud Translation API. For an index builder this site is the clearest example of negative space: there is almost nothing to crawl. The product generates output on demand and stores no public archive — no documents, no posts, no media library, no user-generated corpus. The only crawlable surface is a few hundred near-duplicate marketing and about pages, localized into ~98 UI languages (Probe Log). The thing of value — translation itself — is explicitly walled off from crawlers in robots.txt and offered only through a metered per-character API. inferred Google does not defend Translate against AI crawlers because there is no corpus to protect, only a model sold by the character. Basis: robots.txt (Probe Log) names no AI user agents and instead Disallows the /translate* function paths for every agent, while the Cloud Translation API meters access at $20 per million characters (Verified Metrics).

Retrieval Notes (site-specific deltas)

Only where Translate differs from the default platform assumptions. The largest delta is that there is no corpus, so most "indexing" questions invert into "function-call" questions.

Identifier & dedup

There are no content IDs because there is no content. The only stable identifiers are language codes (ISO-639 / BCP-47, e.g. en, zh-Hans, pt-BR). For the crawlable pages, dedup is the whole problem: ~784 URLs collapse to ~6 canonical pages once you fold ?hl= and /intl/{locale}/ variants.

Retrieval unit

Not a document — a (source text, source lang, target lang) tuple resolved by an API call. If you index anything, you index the capability and the language graph, not pages.

Cross-modal joins

The one meaningful join is language code, which links text↔text across pairs and (if you assemble the pipeline) text↔speech↔image. There is no author, entity, or media graph to join on.

Authoritative content field

No stored field is authoritative; the model output is the answer and it is non-deterministic across model versions. The same input can translate differently after a model update, so any cached translation is a snapshot, not a record.

Freshness & recrawl

The crawlable surface is near-static (recrawl the ~6 about pages monthly). The "data" is infinitely fresh and infinitely ephemeral — generated on demand, never persisted publicly.

Quality & spam

No UGC spam surface. Quality risk is mistranslation and hallucination in low-resource languages — exactly the 110 newest languages where training data is thin (Verified Metrics).

Internal vs external search

There is no search box and no result set; the input box is a function, not a query. Users do not "search Translate," they call it. External indexes can only describe the tool, never mirror its output.

User query profile

Inputs are arbitrary user text/audio/images in any of 243 languages — unbounded and private. None of it is observable to an external indexer.

Indexing Plan

  • Retrieval unit: a capability record, not a page — "Google Translate supports language pair X↔Y via path Z at price P," plus the ~6 canonical about pages as reference docs.
  • Primary keys & joins: language code (ISO-639/BCP-47) as the only stable key; pair = (source code, target code); join the capability record to your own multilingual content by code.
  • Ingestion paths, ranked: 1) Cloud Translation getSupportedLanguages + the languages doc — the language/pair graph as structured reference. 2) Public crawl of the deduped about pages via sitemap.xml. 3) Cloud Translation API as a derivative generator for your own corpus (metered per character), not as a source to harvest. 4) The web translate_a endpoint — disallowed in robots.txt and undocumented; do not build on it.
  • Recrawl cadence: about pages monthly; language graph on Google product announcements (rare, ~annual); pricing page quarterly.
  • Derivatives to generate: if you run your own multilingual index, call the API to translate documents/queries at ingest — store translations as your derivatives under your terms, keyed by language code.
  • Demo to build: a "negative-space census" — crawl the two sitemaps, fold locale variants, and emit the handful of canonical pages plus a language-pair matrix, demonstrating that a top-25 host can have a four-figure URL count and effectively zero indexable corpus.

Risks to the Index

  • Nothing to index is the baseline, not a risk: any indexing strategy that assumes a crawlable corpus here will return near-empty. Budget for API-based derivative generation instead.
  • Per-character metering is the real cost driver: using the API as a derivative generator scales linearly with corpus size at $20/1M chars (NMT) or $10+$10/1M (LLM); a large multilingual index can run a material translation bill.
  • Output is version-dependent: cached translations silently diverge from live output as models update; treat them as dated snapshots, not stable records.
  • inferred The unofficial web endpoint can disappear without notice. Basis: translate_a/single is Disallowed in robots.txt and absent from the Cloud docs (Probe Log); only the documented Cloud Translation API carries a stability/SLA commitment.
