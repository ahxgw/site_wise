Context

microsoft.com is the corporate front door for a $281.7B software-and-cloud company, but as an indexable surface it is mostly two things: a per-product, per-locale marketing layer (the AI, Microsoft 365, Copilot, Security, Dynamics, Power Platform sitemaps) and the Microsoft Store storefront for apps, games, and devices. The dense, high-value Microsoft corpus — technical documentation, support knowledge base, the full app catalog — does not live on this host; it lives on learn.microsoft.com, support.microsoft.com, and apps.microsoft.com, each with its own paths and feeds. The www host is served almost entirely from Akamai edge cache (the homepage is a static blob last modified in 2024; see Probe Log). For an index builder the notable trait is openness: unlike most top-25 sites, Microsoft does not block AI crawlers. inferred Microsoft optimizes microsoft.com for AI discovery rather than walling it. Basis: robots.txt (v2.06, fetched 2026-06-17) carries only User-agent: * path rules plus 23 Sitemap directives — no Disallow for GPTBot, ClaudeBot, Google-Extended, or any named AI agent, consistent with Microsoft owning Bing and Copilot.

Retrieval Notes (site-specific deltas)

Only where microsoft.com differs from the default platform assumptions.

Identifier & dedup

Two ID regimes coexist. Marketing pages key on /{locale}/{product}/... URLs — the same page exists in ~50 locales, so the locale prefix is the dedup axis (collapse to a canonical-language record). Store objects key on a stable 12-character ProductId (e.g. 9NKSQGP7F2NH), independent of URL.

Retrieval unit

The marketing page is the unit for brand/product queries; the Store product (ProductId) is the unit for "download X for Windows" intent. They rarely join — a marketing page for Microsoft 365 is not the Store listing.

Cross-modal joins

Weak on the www host: no JSON-LD means text↔entity joins rely on URL slugs, not declared IDs. The Store side joins cleanly on ProductId across title, publisher, images, ratings, and system requirements.

Authoritative content field

For marketing pages, og:title/og:description are curated and reliable (no SEO keyword stuffing observed). For Store objects, the display-catalog JSON is authoritative over the rendered HTML page.

Freshness & recrawl

Sitemap lastmod values are batch-stamped (an entire locale tree shares one timestamp), so they signal a publishing run, not per-page edits — do not trust them for fine-grained recrawl scheduling.

Quality & spam

Low spam surface: this is first-party corporate content, not UGC. The manipulation risk concentrates in Store reviews, not in the marketing pages.

Internal vs external search

On-site search paths are largely Disallowed (Probe Log); discovery is meant to flow through Bing/Google and through the Store client, not an internal www search box.

User query profile

Navigational and transactional: "download Teams", "Microsoft 365 pricing", "Copilot for business". Users expect to land on an authoritative product or download page, not browse a content corpus.

Indexing Plan

  • Retrieval unit: canonical-language marketing page for brand/product intent; Store product (ProductId) for download/app intent.
  • Primary keys & joins: locale-stripped URL slug for marketing; 12-char ProductId for Store; no reliable key bridging the two.
  • Ingestion paths, ranked: 1) The 23 robots.txt sitemaps with a realistic UA — complete, cheap, no auth, but marketing-only. 2) storeedgefd display-catalog endpoint per ProductId — rich structured + ratings, undocumented so treat as fragile. 3) OG/CDN parsing for images. 4) On-host video and any deep docs — no path here; redirect to learn./support. hosts.
  • Recrawl cadence: marketing pages on a campaign-cycle cadence (weekly sweep; ignore batch lastmod); Store ratings/prices daily; system requirements/metadata on submission change.
  • Derivatives to generate: locale-cluster dedup of marketing pages, ProductId concordance to apps.microsoft.com URLs, OG-image thumbnails.
  • Demo to build: a sitemap-driven crawler that enumerates the 23 product indexes, collapses the ~50 locale variants per page to one canonical record, and reports the true unique-page count per product (closes the Open Question on page count) — with a UA toggle showing the 403↔200 bot-wall flip.

Risks to the Index

  • Store catalog endpoint is undocumented: storeedgefd is the Store client's private API; it can change schema or add auth without notice, breaking the only structured ratings path.
  • Bot wall hardening: the Akamai 403-on-unrealistic-UA behavior (Probe Log) can tighten to challenge-response, cutting off the sitemap crawl even though robots.txt permits it.
  • Corpus lives elsewhere: an index scoped to microsoft.com captures marketing + store but misses the docs/support mass on sibling hosts — a scoping error, not a Microsoft change.
  • inferred AI-friendly stance is a strategic choice, not a stable guarantee. Basis: the open robots.txt (no AI-agent Disallow, observed 2026-06-17) aligns with Microsoft's own Bing/Copilot interests; that alignment, not a public commitment, is what keeps the door open.
