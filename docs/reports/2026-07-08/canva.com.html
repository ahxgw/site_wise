Daily Site Profile

canva.com

Canva is a freemium drag-and-drop design platform whose public crawl surface is almost entirely blocked by a Cloudflare bot challenge, robots.txt notwithstanding.

Design / Creative SaaS

Evidence: verified inferred (must state basis) open — conventions in the methodology.

Context

Canva is a browser and app-based visual design tool: templates, stock media, and an editor for presentations, social posts, video, print, and (since 2024) Affinity's professional desktop suite. Revenue is freemium subscription (Pro, Teams, Enterprise) plus print-on-demand, a creator marketplace for paid templates/elements, and a Connect API integration ecosystem. For an index builder the site is two very different surfaces: a large multilingual help center and marketing shell that is nominally crawlable, and a template/design catalog that robots.txt keeps out of sitemaps entirely — and even the "crawlable" half returns an HTTP 403 challenge to any fetcher without a Cloudflare-trusted fingerprint. inferred Canva's public web is optimized for browser traffic and vetted crawlers, not generic bulk retrieval. Basis: Probe Log — identical 403 challenge on curl, a spoofed Googlebot UA, and Anthropic's own WebFetch tool, while static files under /.well-known/ return 200.

Retrieval Notes (site-specific deltas)

Only where this site differs from the default platform assumptions.

Identifier & dedup

Design/template detail URLs carry a ?uid= query parameter that robots.txt explicitly strips from crawl (both bare and per-locale forms), so the stable per-object ID is deliberately kept out of the crawlable surface rather than just being absent.

Retrieval unit

For the sitemapped half of the site, the help article or category landing page is the unit. For the blocked half (individual templates, editor objects), no public retrieval unit exists outside the Connect API, which is scoped to a single authenticated user's own objects — there is no "crawl all templates" unit available to a third party.

Cross-modal joins

Within the Connect API, design:meta:read and asset:read share the requesting user's account as the join key; there is no public join key between a template's landing-page URL and its underlying design object.

Quality & spam

Not assessed — no bulk template/UGC surface was reachable to sample.

Indexing Plan

The actionable conclusion: how Site Wise would build the index for this site.

  • Retrieval unit: help-center article and category-landing page (only units confirmed reachable via sitemap); template/design objects are not retrievable in bulk by any legitimate path found.
  • Primary keys & joins: URL path as the key for sitemapped pages; no key bridges to individual template/design IDs since those are deliberately excluded from crawl.
  • Ingestion paths, ranked: 1) Sitemap-listed help/marketing URLs, fetched only by a crawler that clears the Cloudflare bot challenge (coverage: ~101.9K URLs, but Site Wise could not itself demonstrate a passing fetch). 2) Connect API via OAuth2 for any Canva account Site Wise controls directly (coverage: that account's own designs/assets only, not the public catalog). 3) Third-party stock APIs (Pexels, Pixabay) for the stock-media subset Canva inherited, run entirely outside Canva's infrastructure.
  • Recrawl cadence: help articles on product-release cadence (unknown, no per-URL lastmod observed); marketing/category pages low-frequency; template catalog freshness unmeasurable from outside.
  • Derivatives to generate: none justified yet — no bulk media object was reachable to justify OCR/ASR/embedding work in this pass.
  • Demo to build: a scripted comparison of response codes across UAs/paths (plain UA, spoofed Googlebot, WebFetch, a real headless browser) against www.canva.com vs. sitemap.canva.com vs. canva.dev, to pin down exactly which layer (IP reputation, TLS fingerprint, JS challenge) is gating access — the open question this report couldn't close with curl/WebFetch alone.

Risks to the Index

Only risks that change indexing strategy.

  • Full-site Cloudflare bot management (observed on every content path tested) means any indexing plan built on "robots.txt allows it" is unreliable without first verifying the fetch actually succeeds — the declared policy and the enforced policy diverge here.
  • Individual template/design pages are excluded from the sitemap by design (the ?uid= disallow rule), so sitemap-based coverage estimates for this site will systematically understate the true catalog and should not be used as a proxy for it.
