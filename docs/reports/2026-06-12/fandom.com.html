Context

Fandom is the web's largest concentration of fan-maintained reference content: 250K MediaWiki instances on *.fandom.com subdomains, one per franchise, plus acquired editorial brands (GameSpot, Metacritic, TV Guide). Founded as Wikia in 2004 by Jimmy Wales and Angela Beesley, rebranded Fandom in 2016, controlled by a TPG-led investor group since 2018. Revenue is display ads layered onto volunteer-written wikis — the tension that drives its biggest structural risk, community forks. For an index builder Fandom is unusual twice over: the text corpus is CC BY-SA licensed, so a full legal mirror is possible; and the access posture is inverted relative to every other platform this project has profiled. The HTML surface — including robots.txt — sits behind a Cloudflare managed challenge, while the per-wiki MediaWiki API, weekly XML dumps on S3, and the image CDN answer anonymous requests with no auth and no quota (Probe Log). The wiki, not the page, is the unit of acquisition: each subdomain is a separate database with its own statistics, dump, sitemap set, and community.

Retrieval Notes (site-specific deltas)

Only where Fandom differs from the default platform assumptions — and it inverts the central one: here the API/dump track is free and unmetered while the HTML track is closed.

Identifier & dedup

Key is (wiki subdomain, pageId); titles move, pageIds don't (probe: Luke Skywalker = pageid 452217). Language editions live as path prefixes on the same subdomain (/es/, /ja/ — 31 on starwars). Rival wikis for one franchise and post-fork stale copies (see Risks) are the real dedup problem, not URL aliases.

Retrieval unit

The article, but flagship-wiki articles are enormous (705KB of wikitext for one character — Probe Log); section-level chunking is mandatory, with the infobox extracted as a separate structured record.

Cross-modal joins

Wikitext file references join articles to CDN image URLs via imageinfo; category membership and interlanguage links join within and across language editions. All keys are MediaWiki-native and stable.

Authoritative content field

Wikitext is ground truth and carries community canon labels (e.g. Wookieepedia's Canon vs Legends) that an index must preserve — they distinguish in-universe fact regimes, not quality tiers.

Freshness & recrawl

Extreme head/tail split: one active wiki exceeds 500 edits/day (Probe Log) while most of the 250K are dormant. recentchanges gives exact per-wiki churn for free — recrawl scheduling can be measured, not guessed.

Quality & spam

Articles-to-pages ratio is a built-in quality denominator (1–31% across the census; the rest is talk, templates, redirects, stubs). Vandalism exists but big wikis revert fast; the bigger trap is fan speculation written in encyclopedic voice.

Internal vs external search

On-site search is weak enough that it is a stated reason communities fork (PC Gamer on the Minecraft vote); discovery runs through Google. An external index with section-level retrieval beats the native experience.

User query profile

Dense entity lookups: character bios, item stats, episode lists, quest walkthroughs, "is X canon". Infobox-shaped answers cover a large share of queries directly.

Indexing Plan

  • Retrieval unit: article section + infobox entity record, keyed to the parent article.
  • Primary keys & joins: (wiki, pageId) with revId for versioning; redirect table for aliases; interlanguage links for cross-edition joins; file references for image joins.
  • Ingestion paths, ranked: 1) S3 XML dumps — full wikitext per wiki, zero auth, refreshed ~weekly on active wikis. 2) api.php recentchanges + parse — real-time incremental on high-churn wikis. 3) wikia.php Discussions — low priority, license unsettled. 4) Image CDN hotlinks for display only. 5) HTML crawl — closed by the Cloudflare challenge; do not fight it, the API track is strictly better anyway.
  • Recrawl cadence: rank wikis by measured recentchanges rate; poll the active head hourly-to-daily via API, sweep the dormant tail from dumps monthly or on dump regeneration.
  • Derivatives to generate: infobox→entity cards, redirect/alias table, cross-wiki and cross-language entity linking, fork-detection signal (edit-rate collapse vs the independent fork), section embeddings.
  • Demo to build: a wiki-census crawler — sweep siteinfo statistics across the subdomain space to produce the per-wiki activity ranking that drives the recrawl schedule, and as a by-product close the "total articles platform-wide" open question.

Risks to the Index

  • Fork exodus poisons the head: when a community forks (Minecraft Wiki left for Weird Gloop's minecraft.wiki on Sep 24 2023), the Fandom copy stays up, keeps its Google rank, and goes stale. An index needs fork detection (edit-rate collapse, fork-notice templates) to switch authority to the successor wiki.
  • The open API track is one config flip from closing: the Cloudflare wall already covers all HTML; extending it to api.php and the S3 bucket requires no engineering. Mirror dumps while they're open — CC BY-SA makes the mirror lawful and irrevocable for text already published.
  • AI-crawler policy is selective today, expandable tomorrow: robots.txt bans only GPTBot and ImagesiftBot among AI agents (Probe Log) — unusually permissive vs the default assumption — but the same file is one deploy away from the industry-standard blanket ban.
  • Image rights are the weak layer: text is CC but the image corpus is largely fair-use franchise media; any index feature beyond hotlink display needs a rights review per use, not per site.

Inferences

  • Fandom's accounts are one global pool, so the siteinfo users figure is platform-wide, not per-wiki. Basis: near-identical counts (41,274,088 vs 41,276,186) returned by two unrelated wikis probed minutes apart on 2026-06-12.
  • Approved search crawlers pass the Cloudflare wall that blocks everyone else. Basis: cf-mitigated: challenge on every client we tried, while robots.txt maintains 31 sitemap directives and current fandom.com pages keep appearing in live search results observed 2026-06-12 — pure SEO plumbing nobody could use if Google were challenged too.
  • The open API/dump posture is deliberate policy, not an oversight. Basis: robots.txt explicitly Allows api.php for all agents while Disallowing utility namespaces, and the dump pipeline is documented user-facing functionality (Help:Database_download).
