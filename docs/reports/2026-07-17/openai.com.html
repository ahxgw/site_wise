Context

openai.com is the corporate, research, and product-marketing site of OpenAI, developer of ChatGPT and the GPT model family. The nonprofit OpenAI Foundation controls OpenAI Group PBC, the public-benefit for-profit subsidiary that runs the commercial business; the conversion completed Oct 28 2025, with Microsoft taking a 27% stake. Revenue comes from ChatGPT subscriptions plus metered API usage; the company closed a $122B funding round at an $852B valuation on Mar 31 2026 and reported $25B+ in annualized revenue as of end-Feb 2026. For an index builder, this host is almost entirely unreachable by ordinary automated fetch: Cloudflare's bot-management layer returned a 403 challenge for every sampled HTML page — homepage and article alike — regardless of user agent, while robots.txt, sitemap.xml, and a 1,038-item RSS feed stayed open. inferred That split looks like a deliberate content-type policy rather than a network fluke. Basis: four fetch attempts across two URLs and three methods (default-UA curl, Googlebot-UA curl, WebFetch) all returned identical cf-mitigated: challenge headers, while every non-HTML asset path tried (robots.txt, sitemap.xml, RSS, bot-IP JSON) returned 200 (Probe Log).

Retrieval Notes (site-specific deltas)

Only where this site differs from the default platform assumptions.

Identifier & dedup

Canonical key is the /index/{slug}/ path, shared by the sitemap <loc> and the RSS <guid>. Locale variants appear as /{locale}/index/{slug}/ via xhtml:link alternates in the sitemap (8+ locales seen in one sampled entry, e.g. de-DE, id-ID, el-GR, zh-Hant, pl-PL); strip the locale prefix to reach the canonical page.

Retrieval unit

The individual post at /index/{slug}/. Its metadata (sitemap lastmod + locale alternates, RSS title/description/category/pubDate) is retrievable; the body text is not, since every page-render fetch this session hit the Cloudflare challenge.

Cross-modal joins

RSS <guid> equals the canonical (non-locale) sitemap <loc> — the two feeds join on exact URL. No image/video join key was confirmed (Open Questions).

Authoritative content field

RSS <description> is a one-sentence marketing summary, not the article body — even the accessible channel yields metadata, not full text (Probe Log RSS sample).

Freshness & recrawl

RSS is append-only and same-day fresh; a baseline crawl of the 1,038 existing items plus daily RSS polling captures every new post with no gap, independent of the blocked HTML layer.

Quality & spam

Single-publisher corporate/research blog, no UGC on this host — no spam-manipulation surface here (Access Matrix).

Indexing Plan

  • Retrieval unit: RSS item / sitemap URL, keyed by the canonical (non-locale) /index/{slug}/ path.
  • Primary keys & joins: canonical path stripped of any /{locale}/ prefix; RSS <guid> matches sitemap <loc> exactly.
  • Ingestion paths, ranked: 1) Poll news/rss.xml (unauthenticated, 1,038-item full archive, same-day updates) for title/description/category/pubDate — the only channel that isn't CDN-challenged. 2) Sitemap-diff the 34 child sitemaps for lastmod-based change detection and locale-alternate discovery, without rendering the page. 3) Full-text ingestion of the article body has no observed legitimate path this session — every direct page fetch (curl, spoofed-UA curl, WebFetch) hit Cloudflare's challenge; the only realistic route is a general search engine's own cache/snippet, not a first-party crawl.
  • Recrawl cadence: RSS/sitemap poll daily (new items publish same-day); no cadence applies to body text since it isn't ingestible through any path found this session.
  • Derivatives to generate: none justified beyond embedding the RSS description sentence itself for topic classification against the existing 19-tag taxonomy as labels — there is no body text to run ASR/OCR against yet.
  • Demo to build: an RSS-plus-sitemap join watcher — poll rss.xml, match each item's guid against the sitemap's locale-alternate list, and report "this post has N locale variants, published/updated on DATE" purely from metadata, never touching the challenged HTML endpoint.

Risks to the Index

  • CDN-layer bot wall, not a robots.txt policy: a Cloudflare bot-management challenge blocked 100% of sampled HTML page fetches (2 curl UAs + WebFetch, across 2 URLs) while robots.txt/sitemap.xml/RSS/bot-IP-JSON stayed open — a far harder gate than a disallow rule, since a well-behaved crawler has nothing in robots.txt to respect or negotiate around (Probe Log).
  • inferred The HTML/asset split looks deliberate, not incidental. Basis: 4 independent fetch attempts across 2 URLs and 3 methods all returned identical cf-mitigated: challenge headers, while every non-HTML path tried returned 200 (Probe Log).
  • Revenue/valuation figures are third-party-attributed: $25B ARR traces to The Information via Reuters, not an openai.com disclosure page reachable this session; a post-IPO 10-K would be a more authoritative but structurally different source to re-point citations to.
  • Reciprocity asymmetry: OpenAI operates three named crawlers (GPTBot, OAI-SearchBot, OAI-AdsBot) that other sites choose to allow or block via robots.txt, while openai.com's own CDN blocks equivalent inbound automated fetches outright — an indexer relying on crawler reciprocity norms gets no default access here.
