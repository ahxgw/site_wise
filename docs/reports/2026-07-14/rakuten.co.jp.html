Context

Rakuten Ichiba is a shop-centric marketplace, not a retailer: merchants (~50,000+, third-party marketing pages) run their own storefronts, pricing, and creative, and pay Rakuten commission and ad fees rather than selling inventory to it. That structure makes Rakuten closer to eBay or Shopify than to Amazon. The mall is one of 70+ businesses under Rakuten Group's single membership ID, cross-subsidized by FinTech (card, bank, securities, insurance) and Mobile segments and bound together by Rakuten Super Points, a shared loyalty currency. For an index builder the marketplace exposes almost no on-page schema.org markup (Probe Log) and pushes structured access toward a metered, application-ID-gated API rather than a sitemap-driven crawl — the opposite tradeoff from a typical SEO-optimized storefront.

Retrieval Notes (site-specific deltas)

Identifier & dedup

itemCode is scoped per shop (shopCode_itemId), not a single global product ID like Amazon's ASIN — the same physical product listed by two shops gets two unrelated item codes, so cross-shop dedup needs title/JAN/ISBN matching rather than an ID join.

Retrieval unit

The shop-item pair is the natural unit (one storefront's listing), not a canonical product; a "product" concept spanning shops doesn't exist in Rakuten's own data model the way ASIN variation families do for Amazon.

Authoritative content field

No schema.org Product/Offer markup exists on either the mall or books verticals (Probe Log) — the RWS API response, not the rendered page, is the closest thing to ground truth for price and availability.

Freshness & recrawl

The books vertical's sitemap carries lastmod timestamps and is the only Ichiba-family property with a declared crawl surface at all; the core mall (item/search/ranking) has none, pushing any recrawl strategy for it entirely onto the metered API rather than sitemap polling.

Quality & spam

Because storefronts (not Rakuten) write titles and catchcopy, keyword-stuffed titles and duplicate near-identical shop-item pairs for the same product are a structural pattern, not an edge case — more pronounced than on inventory-owned retailers.

Internal vs external search

The generic-UA-friendly item pages (Probe Log) suggest external search engines can index item text more easily here than on Amazon; but with no Product schema, an external index has no structured price/availability signal to rank on without hitting the RWS API separately.

Indexing Plan

  • Retrieval unit: shop-item pair (shopCode + itemCode); rating summary attached as a fast-refresh sub-record; genre node attached for category context.
  • Primary keys & joins: shopCode+itemCode as primary key; JAN/ISBN (where present in item text) as a soft cross-shop join key, since no hard cross-shop product ID exists.
  • Ingestion paths, ranked: 1) RWS IchibaItem Search/Ranking APIs — metadata, thumbnails, rating summary; capped at 1 req/sec per Application ID with no relaxation. 2) Direct HTML fetch of item.rakuten.co.jp pages under a plain UA for full body text and higher-res images, where the API's small/medium thumbnails fall short — fragile, since UA/Referer combinations that work today may not tomorrow (Probe Log). 3) books.rakuten.co.jp sitemap — the one Ichiba-family property with a declared, incrementally crawlable URL list. 4) Review text — no legitimate path; design around count/average only.
  • Recrawl cadence: rankings via the Ranking API's realtime option where genre coverage matters; item metadata weekly; books sitemap re-diffed on its own lastmod values.
  • Derivatives to generate: JAN/ISBN extraction from item text for cross-shop and cross-site product matching; near-duplicate detection across shop-item pairs selling the same physical good.
  • Demo to build: a quota-aware RWS indexer — take a genre ID, walk the Ranking API within the 1 req/sec budget, emit JSONL records, and surface which fields (review text, high-res images, cross-shop product identity) the API structurally can't supply.

Risks to the Index

  • No declared crawl surface for the core mall: item/search/ranking robots.txt carry no Sitemap directive (Probe Log); an index built by direct HTML fetch has no official discovery path and depends on inbound links or the RWS API's own search/ranking calls to enumerate items.
  • Bot mitigation is inconsistent per property, not centrally enforced: opposite UA polarities trip blocks on item.rakuten.co.jp vs. books.rakuten.co.jp (Probe Log) — a scraper tuned against one Rakuten property can silently start failing on another.
  • inferred Review text is permanently API-excluded, not temporarily withheld. Basis: the RWS Item Search field list ships reviewCount/reviewAverage only (no text field to later populate), and review.rakuten.co.jp itself 403s at the gateway layer (Probe Log) — mirrors Amazon's PA-API pattern.
  • FY2025 net loss is the 7th consecutive year (Verified Metrics); a conglomerate under sustained financial pressure is a plausible candidate for further API/rate-limit tightening or new paid tiers, though nothing in the current docs signals an imminent change.
