Context

ESPN.com is the United States' dominant sports destination: news, scores, standings, and — since August 2025 — a standalone streaming product that put ESPN's cable networks online without a pay-TV bundle for the first time. The business is carriage-and-advertising media (Disney's Sports segment, $17.7B FY2025 revenue) pivoting to direct subscriptions. For an index builder, the striking fact is architectural: ESPN's editorial HTML and live video are defended by a CloudFront bot wall (Probe Log), yet the same site is backed by a family of undocumented but wide-open JSON APIs (site.api.espn.com, sports.core.api.espn.com) that return the full sports entity graph — leagues, seasons, events, plays, teams, athletes, standings — with no key, no rate limit observed, and Access-Control-Allow-Origin: *. The indexable object here is not the page; it is the event and the entity graph hung off it. inferred ESPN treats structured sports data as a display feed, not a protected asset. Basis: the open CORS API and 6-second scoreboard cache observed in the Probe Log, versus the 403 wall on www.espn.com HTML and sitemap.xml.

Retrieval Notes (site-specific deltas)

Only where ESPN differs from the default platform assumptions.

Identifier & dedup

ESPN's IDs are unusually clean: a compact uid encodes the hierarchy (s:20~l:28~t:{team}~a:{athlete}), and every entity resolves through a self-describing $ref URL. Sport + league + event + competition uniquely key a game; the same athlete carries one ID across every season and team.

Retrieval unit

The event (a single game) is the natural unit: it joins competitors, venue, plays, box score, odds, and linked articles. For editorial, the unit is the article keyed by contentKey/id.

Cross-modal joins

Everything joins on the entity IDs: an article's categories[] carry team/athlete/league IDs, so a story links to the same athlete node the box score references. Images join by the same IDs. This is a rare site where text and structured data share one key space.

Authoritative content field

For facts (scores, rosters, standings) the API is ground truth and updates faster than the prose. Article bodies are commentary; the structured feed is the record. When a headline and the scoreboard disagree, trust the scoreboard.

Freshness & recrawl

Two clocks: live scoreboards carry a 6-second cache and must be polled during games; historical seasons never change. Recrawl by event state (pre/in/post), not by a site-wide cadence.

Internal vs external search

The bot wall on HTML plus a blocked sitemap means external crawlers cannot see article bodies at all; only the API's headline/description is reachable. External text indexes of ESPN are structurally thin — the structured graph is the only fully open surface.

Quality & spam

Near-zero UGC spam surface: the corpus is first-party editorial and league-sourced stats. Risk is staleness (a missed live poll), not manipulation.

User query profile

Score checks ("did the Lakers win"), schedule/standings lookups, player stat lines, and fantasy research — all answerable from the structured feed without touching an article.

Indexing Plan

  • Retrieval unit: the event (game) with its entity graph attached; the article as a secondary unit keyed by contentKey.
  • Primary keys & joins: the uid hierarchy (sport~league~team~athlete) and numeric event/athlete/team IDs; articles join to entities through categories[].
  • Ingestion paths, ranked: 1) Core API (sports.core.api.espn.com) — walk league→season→events→athletes via $ref; pagination totals give exact counts. 2) Site API (site.api.espn.com) — scoreboards, news headlines, images. 3) Article bodies — no legitimate bulk path; HTML is 403-walled, sitemap blocked. 4) Video — subscription only, not indexable.
  • Recrawl cadence: live events every ~10s while in-progress; standings/rosters daily; historical seasons once.
  • Derivatives to generate: an entity concordance (ESPN IDs ↔ league/Sportradar IDs), event-to-article link graph, headshot/logo embeddings for visual entity resolution.
  • Demo to build: an event-graph crawler that walks the Core API $ref chain for one league, emits JSONL entity docs with counts, and flags every field that has no documented reuse license.

Risks to the Index

  • The open API is undocumented, not licensed. It has no published terms and no SLA; Disney's Terms of Use prohibit scraping and commercial reuse. Access can be keyed, throttled, or shut off without notice — build for that.
  • Legal exposure is the real wall, not the technical one. The data flows freely, but redistributing scores/stats commercially invites both ESPN's terms and league data-rights claims (Sportradar/Genius licensing). Display rights, not fetch access, are the constraint.
  • inferred The AI-crawler block list is stale and partial. Basis: robots.txt (observed 2026-07-02) blocks GPTBot, anthropic-ai, CCBot, and Bytespider but omits ClaudeBot, PerplexityBot, Applebot-Extended, and Amazonbot — enforcement lags the current agent landscape.
  • Video and premium editorial are moving behind the DTC paywall. The Aug 2025 launch and the premium flag on articles signal more content gated over time; the free structured surface may narrow.
